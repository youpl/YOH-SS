	##  Préservation & contexte (AVANT TOUT)
	Objectif :
		* Éviter la destruction de preuves
		* Fixer le contexte temporel

	Étapes :
		* Noter l’heure exacte de début de vérification
		* Noter l’heure de connexion au serveur
		* Noter l’heure estimée de lancement de Minecraft
		* ⚠️ Ne PAS redémarrer le PC
		* ⚠️ Ne PAS fermer les applications en cours
		* ⚠️ Ne PAS lancer de scans destructifs (cleaners, antivirus)

	⚠️ Toute action ultérieure sera corrélée à ces horaires
	
	RÈGLE D’OR FORENSIQUE
		* Un indice seul = JAMAIS suffisant
		* Une corrélation temporelle = preuve
		* USN + Prefetch + SRUM = combo gagnant
		* Le but n’est PAS de prouver COMMENT,
		  mais QUAND et QUOI

	##	Mémoire & exécution ACTIVE 
		Objectif :
			* Identifier ce qui est ACTUELLEMENT en mémoire
			* Détecter un cheat actif ou un loader encore présent
			* Capturer les informations utiles AVANT que le processus ne se ferme
			* Repérer les processus parents / enfants liés à Minecraft
			* Orienter rapidement vers USN, Prefetch, SRUM ou analyse mémoire

		Étapes :
			1. Préservation immédiate
				* Noter l'heure exacte du début de l'analyse
				* NE PAS fermer Minecraft
				* NE PAS tuer de processus suspect avant capture des informations
				* Si un processus suspect est visible :
					→ prendre une capture d'écran
					→ noter son nom
					→ noter son PID
					→ noter son chemin si visible
					→ noter l'heure de détection

			2. Ouvrir le Gestionnaire de tâches
				* Windows + R taper :
					taskmgr
				⚠️ Si Gestionnaire désactivé 					→ Voir ANNEXE 1.a
				⚠️ Si erreur "Cette opération a été annulée" 	→ Voir ANNEXE 1.b

				* Aller dans l'onglet "Détails"
				* Clic droit sur les colonnes -> Sélectionner les colonnes
				* Ajouter si disponible :
					- PID
					- Nom d'utilisateur
					- Ligne de commande
					- Description
					- Éditeur / Publisher

			3. Analyser les processus actifs
				* Trier par Mémoire décroissante + -> -
				* Trier aussi par CPU si besoin
				* Identifier :
					- javaw.exe
					- java.exe
					- processus inconnus
					- noms aléatoires
					- processus sans icône / éditeur
					- processus lancés depuis AppData / Temp / Downloads / Desktop / OneDrive
					- cmd.exe / powershell.exe / wscript.exe / cscript.exe / mshta.exe
					- rundll32.exe / regsvr32.exe si suspicion de DLL
					- AutoHotkey.exe / python.exe / node.exe selon contexte

				* Pour chaque processus suspect :
					→ clic droit -> Ouvrir l'emplacement du fichier
					→ clic droit -> Propriétés
					→ vérifier signature / éditeur / date de création
					→ NE PAS exécuter le fichier manuellement

			4. Extraction rapide PowerShell
				* Windows + X -> Terminal / PowerShell (Administrateur)
				* Lister les processus actifs avec chemin et ligne de commande :
					Get-CimInstance Win32_Process | Select-Object ProcessId, ParentProcessId, Name, ExecutablePath, CommandLine | Sort-Object Name

				* Filtrer les processus suspects dans les dossiers utilisateur :
					Get-CimInstance Win32_Process | Where-Object { $_.ExecutablePath -match '\\Users\\|\\AppData\\|\\Temp\\|\\Downloads\\|\\Desktop\\|\\OneDrive' } | Select-Object ProcessId, ParentProcessId, Name, ExecutablePath, CommandLine

				* Filtrer les interpréteurs / lanceurs à risque :
					Get-CimInstance Win32_Process | Where-Object { $_.Name -match 'java|cmd|powershell|wscript|cscript|mshta|rundll32|regsvr32|python|node|autohotkey' } | Select-Object ProcessId, ParentProcessId, Name, ExecutablePath, CommandLine

				* Si un chemin suspect est trouvé, calculer le hash :
					Get-FileHash -Path "C:\chemin\vers\fichier.exe" -Algorithm SHA256

				* Vérifier la signature :
					Get-AuthenticodeSignature -FilePath "C:\chemin\vers\fichier.exe"

			5. Analyse avancée avec Process Explorer / System Informer
				* Ouvrir Process Explorer ou System Informer en Administrateur
				* Activer l'affichage détaillé :
					- Process Explorer : File -> Show Details for All Processes
					- System Informer : Hacker/System -> Show Details for All Processes

				* Observer :
					- arbre parent / enfant
					- chemin complet
					- ligne de commande
					- signature numérique
					- processus sans éditeur
					- processus lancés juste avant / pendant Minecraft

				* Sur un processus suspect :
					→ clic droit -> Properties
					→ noter Path / Command Line / Parent / Start Time
					→ onglet Modules / DLLs si disponible
					→ chercher des DLLs depuis AppData / Temp / Downloads

				⚠️ Si une DLL suspecte est chargée dans javaw.exe :
					→ noter son chemin
					→ corréler avec Phase 4.6, Phase 5 et recherche DLL

			6. Vérifier les applications de démarrage / persistence rapide
				* Gestionnaire de tâches -> Applications de démarrage
					- tout exe non système = suspect
					- tout chemin AppData / Temp / Downloads = suspect

				* Vérifier les dossiers de démarrage :
					Windows + R :
						shell:startup
					Windows + R :
						shell:common startup

				* Vérifier les clés Run :
					Get-ItemProperty 'HKCU:\Software\Microsoft\Windows\CurrentVersion\Run'
					Get-ItemProperty 'HKLM:\Software\Microsoft\Windows\CurrentVersion\Run'

				* Vérifier les tâches planifiées non Microsoft :
					Get-ScheduledTask | Where-Object { $_.TaskPath -notlike '\Microsoft*' } | Select-Object TaskName, TaskPath, State

			7. Interprétation rapide
				* Processus suspect actif + chemin utilisateur
					→ priorité haute
				* Processus suspect actif + Minecraft ouvert
					→ corrélation temporelle forte
				* Processus sans signature + nom aléatoire
					→ suspect
				* Processus disparu pendant l'analyse
					→ Voir ANNEXE 1.d puis corréler USN / Prefetch / SRUM
				* javaw.exe lancé depuis un chemin non attendu
					→ vérifier Phase 2
				* rundll32.exe proche d'une DLL suspecte
					→ vérifier Phase DLL / USN / Prefetch

			⚠️ PRIORITÉ ABSOLUE  
			Si un cheat est visible ici :
				→ capture d'écran
				→ PID
				→ chemin
				→ ligne de commande
				→ hash
				→ corrélation avec heure de jeu
				= preuve directe très forte

		##	Exécution Java & cheats .jar
		Objectif :
			* Détecter les autoclickers / cheats Java externes
			* Identifier un .jar lancé via java.exe / javaw.exe
			* Repérer les traces JNativeHook / hooks clavier-souris
			* Distinguer le Java légitime de Minecraft d'un Java externe
			* Corréler les .jar avec USN, Prefetch, SRUM et mémoire

		Rappel important :
			* Un cheat .jar peut être lancé par java.exe / javaw.exe
			* Le Prefetch ne crée généralement PAS un fichier .pf au nom du .jar
			* La preuve d'exécution passe souvent par :
				→ ligne de commande Java contenant -jar
				→ chemin du .jar dans la ligne de commande
				→ trace USN du .jar
				→ Prefetch JAVA.EXE / JAVAW.EXE
				→ SRUM montrant java/javaw actif pendant Minecraft

		Étape 1 | Recherche JNativeHook / traces Java temporaires
			1. Vérifier graphiquement les fichiers temporaires
				* Windows + R :
					%temp%
				* Rechercher les mots-clés :
					- JNativeHook
					- jnativehook
					- jna
					- java
					- .jar
					- .dll

			2. Vérifier via PowerShell
				* Windows + X -> Terminal / PowerShell (Administrateur)
				* Taper la commande :
					Get-ChildItem $env:TEMP -Recurse -Force -ErrorAction SilentlyContinue | Where-Object { $_.FullName -match 'JNativeHook|jnativehook|jna|java|\.jar$|\.dll$' } | Select-Object FullName, LastWriteTime, Length | Sort-Object LastWriteTime -Descending

			3. Interpréter :
				* JNativeHook récent proche de l'heure de jeu
					→ suspicion d'autoclicker / macro Java
				* .dll Java/JNA dans Temp
					→ possible bibliothèque native extraite par un programme Java
				* .jar récent dans Temp / Downloads / Desktop
					→ corréler avec Phase 4.6 et Phase 5

			⚠️ JNativeHook seul n'est PAS une preuve suffisante.
			⚠️ JNativeHook + javaw actif + activité pendant Minecraft = signal fort.

		Étape 2 | Lister les processus Java actifs
			1. Lister Java avec chemin + ligne de commande
				* Windows + X -> Terminal / PowerShell (Administrateur)
				* Taper la commande :
					Get-CimInstance Win32_Process | Where-Object { $_.Name -match '^javaw?\.exe$' } | Select-Object ProcessId, ParentProcessId, Name, ExecutablePath, CommandLine, CreationDate

			2. Exporter si besoin pour conservation :
				Get-CimInstance Win32_Process | Where-Object { $_.Name -match '^javaw?\.exe$' } | Select-Object ProcessId, ParentProcessId, Name, ExecutablePath, CommandLine, CreationDate | Out-File "$env:USERPROFILE\Desktop\java_processes.txt"

			3. Points à regarder :
				* ProcessId / PID
					→ permet de retrouver le même processus dans System Informer
				* ParentProcessId
					→ permet de savoir quel launcher / programme a lancé Java
				* ExecutablePath
					→ emplacement du java.exe / javaw.exe
				* CommandLine
					→ contient souvent -jar, -cp, -classpath ou le chemin du .jar
				* CreationDate
					→ heure de lancement à comparer avec Minecraft

			⚠️ Si la ligne de commande est vide ou inaccessible		→ Voir ANNEXE 2.a

		Étape 3 | Interpréter les chemins Java
			* Chemins Minecraft / launchers possibles :
				- C:\Users\<Utilisateur>\AppData\Roaming\.minecraft\
				- C:\Users\<Utilisateur>\AppData\Roaming\.minecraft\runtime\
				- C:\Users\<Utilisateur>\AppData\Roaming\.minecraft\jre\
				- C:\Program Files (x86)\Minecraft Launcher\
				- dossiers de launchers connus : Lunar, Badlion, Feather, Labymod, Modrinth, Prism, MultiMC

			* Chemins suspects :
				- C:\Users\<Utilisateur>\Downloads\
				- C:\Users\<Utilisateur>\Desktop\
				- C:\Users\<Utilisateur>\AppData\Local\Temp\
				- C:\Users\<Utilisateur>\AppData\Roaming\ avec nom inconnu
				- dossier au nom aléatoire

			⚠️ Plusieurs launchers = plusieurs javaw possibles.
			⚠️ Un Java hors `.minecraft` n'est pas automatiquement un cheat.
			⚠️ Ce qui compte : chemin + ligne de commande + heure + corrélation.

		Étape 4 | Détecter un .jar via la ligne de commande
			* Rechercher dans la sortie PowerShell :
				- -jar
				- .jar
				- -cp
				- -classpath
				- @C:\... (fichier d'arguments Java)

			* Interprétation :
				* CommandLine contient `-jar C:\...\nom.jar`
					→ exécution directe d'un .jar
				* CommandLine contient `-cp` ou `-classpath` avec un .jar externe
					→ Java charge un .jar via classpath
				* CommandLine contient un fichier `@arguments`
					→ ouvrir le fichier d'arguments si le chemin est visible et vérifier les .jar chargés
				* CommandLine pointe vers Downloads / Temp / Desktop
					→ suspect si proche de l'heure de jeu

	## USN JOURNAL (Méthode Semi automatique)	
	Cette méthode est utilisée afin d'identifier des exécutables / scripts / logs supprimés ou renommés, même si le joueur tente de nettoyer ses traces.
	⚠️ Le USN Journal ne prouve PAS l'injection mémoire, mais il prouve des actions disque (création, renommage, suppression, troncature).

	Objectif de la méthode USN
			* Détecter des .exe créés puis supprimés (cheat portable / loader)
			* Détecter des renommages visant à dissimuler un fichier
			* Détecter des suppressions ciblées de Prefetch (.pf) / logs (.evtx)
			* Détecter des comportements anti-forensic (troncature, object id change)
			* Construire une timeline “CREATE → (RENAME) → DELETE” corrélable avec Minecraft
			
	Pré-requis / Précautions
		* Windows + X → Terminal / PowerShell (Administrateur)
		* NE PAS redémarrer le PC
		* NE PAS lancer de “cleaner”
		* Noter l’heure exacte du début d’analyse (utile pour filtrer les derniers événements)


	Vérification préalable du USN Journal
		* Taper la commande : 
			fsutil usn queryjournal C:
		* Interpréter le résultat :
			* Maximum Size 		→ 	Taille max du journal (défaut ~32-128 MB)
			* Allocation Delta	→	Incrément d'allocation
			* USN Journal ID		→ 	Identifiant unique du journal
		⚠️ Si "USN Journal is disabled"	Le journal n'existe pas
			⚠️Suspect si désactivé manuellement (possible nettoyage de traces) :
				→ Passer directement à la Phase 5 - PREFETCH
			✅ Si actif :
				→ Continuer ci-dessous

	Analyse automatique (recommandée)
		Taper la commande :
			$e='\.(ahk|exe|bat|dll|ps1|zip|rar|jar|py|vbs|vbe|wsh|psm1|pyc|pl|sh|cpl)'; $x='__PSScriptPolicyTest|BACKGROUNDTASKHOST|MOUSOCOREWORKER|SVCHOST|RUNTIMEBROKER|TASKHOSTW|SPPSVC|CONSENT|WT|SMARTSCREEN|OPENCONSOLE|WINDOWSTERMINAL|FSUTIL|DLLHOST|EXPLORER|TIWORKER|SDBINST|UPDATER|CONHOST|STEAMWEBHELPER'; $Desktop=[Environment]::GetFolderPath("Desktop"); New-Item -Path "$Desktop\VerifUSN" -ItemType Directory -Force|Out-Null; $output="$Desktop\VerifUSN\usn_suspects.txt"; Write-Host "`n[1/2] Comptage des entrées USN..." -ForegroundColor Cyan; $total=(fsutil usn readjournal C: | Measure-Object -Line).Lines; Write-Host "[OK] $total entrées totales`n[2/2] Analyse..." -ForegroundColor Green; "=== USN SCAN - $(Get-Date -Format 'yyyy-MM-dd HH:mm:ss') ===`n=== TOTAL: $total entrées ===`n"|Out-File $output; $b=@(); $n=0; $i=0; fsutil usn readjournal C: | % { $i++; if($i % 1000 -eq 0){Write-Progress -Activity "USN Analysis" -Status "$([math]::Round($i/$total*100,1))% | Suspects: $n" -PercentComplete ($i/$total*100)}; if($_ -match '^USN'){ $t=$b -join "`n"; if($t -match $e -and $t -notmatch $x){$b|Out-File -Append $output; ""|Out-File -Append $output; $n++}; $b=@()}; $b+=$_}; Write-Progress -Completed -Activity "USN"; Write-Host "`n✓ $n suspects ($([math]::Round($n/$total*100,2))%) -> $output`n" -ForegroundColor Green
		* Durée estimée : 2-15 minutes selon la taille du journal
		* Résultat :
		* Fichier créé : Bureau\VerifUSN\usn_suspects.txt
			* Contient uniquement les événements suspects filtrés
			* Affiche progression en temps réel
		⚠️ Dans le cas ou la commande ne fonctionne pas il est possible de faire cette vérification manuellement voir la "Phase 4.5"

			echo ===== Fichier dll Crée Récemment===== > dllActivity.txt && fsutil usn readjournal C: csv | findstr /i /c:".dll" | findstr /i /c:0x00000100 >> dllActivity.txt && echo. >> dllActivity.txt && echo ===== Fichier dll Supprimer Récemment ===== >> dllActivity.txt && fsutil usn readjournal C: csv | findstr /i /c:".dll" | findstr /i /c:0x80000200 >> dllActivity.txt && echo. >> dllActivity.txt && echo ===== Fichier dll Renommer Récemment ===== >> dllActivity.txt && fsutil usn readjournal C: csv | findstr /i /c:".dll" | findstr /i /c:0x00001000 >> dllActivity.txt && echo. >> dllActivity.txt && echo ===== Fichier dll Renommer Précédement ===== >> dllActivity.txt && fsutil usn readjournal C: csv | findstr /i /c:".dll" | findstr /i /c:0x00002000 >> dllActivity.txt && notepad dllActivity.txt

	Étape 4 | Analyse des résultats
		* Regarder en priorité :
			- fichiers dans AppData / Temp / Downloads / Desktop / OneDrive
			- noms aléatoires ou trompeurs
			- .jar créé ou supprimé proche de l'heure de jeu
			- .exe créé puis supprimé en quelques minutes
			- .bat supprimé après lancement d'un .exe ou .jar
			- .dll créée récemment dans System32, Temp, AppData ou dossier inconnu
			- renommage .tmp -> .dll / .txt -> .jar / .png -> .exe

		* Interprétation :
			* FILE_CREATE + FILE_DELETE proche dans le temps
				→ fichier probablement utilisé puis nettoyé
			* RENAME_OLD_NAME puis RENAME_NEW_NAME
				→ vérifier l'ancien et le nouveau nom
			* .dll présente dans USN mais absente du disque
				→ possible injection suivie d'un nettoyage
			* .bat supprimé après activité .exe/.jar
				→ possible script de lancement ou nettoyage

		⚠️ USN seul ne prouve PAS l'exécution.
		⚠️ Pour conclure, corréler avec :
			- Phase 5 - PREFETCH
			- Phase 10 - SRUM
			- Windows.db / Historique des Applications
			- Analyse mémoire Javaw / System Informer

	Phase 5 – PREFETCH (preuve d’exécution)

		Objectif :
			* Prouver qu’un programme a été exécuté sur le système
			* Retrouver l'heure de dernière exécution d'un loader / autoclicker / script
			* Confirmer un fichier suspect trouvé en USN Journal
			* Identifier des exécutables supprimés après utilisation
			* Corréler l'exécution avec l'heure de lancement de Minecraft

		Outils requis :
			* WinPrefetchView					→ Voir Tools Bible
			* PECmd si besoin d'un export CSV avancé	→ Voir Tools Bible
			* PowerShell / CMD Administrateur

		Rappel important :
			* Un fichier Prefetch (.pf) est créé / mis à jour quand Windows exécute un programme
			* Le Prefetch prouve l'exécution d'un .exe, même si l'exécutable original a été supprimé
			* Le Prefetch ne prouve PAS directement une injection dans javaw.exe
			* Le Prefetch ne trace pas directement les .jar, mais peut prouver l'exécution de javaw.exe / java.exe ou d'un loader qui lance le .jar

		Étape 1 | Vérification rapide du dossier Prefetch
			* Windows + X -> Terminal / PowerShell (Administrateur)
			* Taper la commande :
				Get-ChildItem -Path 'C:\Windows\Prefetch' -Filter *.pf | Select-Object Name, LastWriteTime, Length | Sort-Object LastWriteTime -Descending

			* Identifier :
				- fichiers .pf très récents
				- noms aléatoires
				- loaders / autoclickers connus
				- CMD.EXE / POWERSHELL.EXE / WSCRIPT.EXE / CSCRIPT.EXE / MSHTA.EXE
				- RUNDLL32.EXE proche d'une activité .dll suspecte
				- JAVA.EXE / JAVAW.EXE proche d'une activité .jar suspecte

			* Colonnes utiles :
				* Name			→ nom du fichier Prefetch
				* LastWriteTime	→ dernière mise à jour du Prefetch, donc exécution récente
				* Length		→ taille du fichier Prefetch, utile comme indice d'activité

		Étape 2 | Analyse avec WinPrefetchView (recommandée)
			* Télécharger / ouvrir WinPrefetchView
				* Lien : https://www.nirsoft.net/utils/trans/winprefetchview_french.zip
				* Lancer WinPrefetchView.exe en Administrateur

			* Chargement des Prefetch :
				* WinPrefetchView charge généralement automatiquement :
					C:\Windows\Prefetch
				* Si rien ne s'affiche :
					→ vérifier les droits administrateur
					→ vérifier que le dossier C:\Windows\Prefetch contient des fichiers .pf
					→ si le dossier est vide ou presque vide, suspecter nettoyage / Prefetch désactivé
					→ Voir ANNEXE 5.a

			* Trier les résultats :
				* Trier par "Dernière exécution" / "Last Run Time" décroissant
				* Regarder les exécutions proches de :
					- heure de lancement Minecraft
					- heure de connexion serveur
					- heure des traces USN .exe / .bat / .dll / .jar

			* Colonnes à analyser :
				* Filename / Nom du Prefetch
					→ nom du .pf créé par Windows
				* Process EXE / Exécutable
					→ nom réel du programme exécuté
				* Process Path / Chemin du processus
					→ emplacement du programme si disponible
				* Run Counter / Nombre d'exécutions
					→ nombre de lancements enregistrés
				* Last Run Time / Dernière exécution
					→ horaire principal à corréler avec Minecraft
				* Missing Process / Processus manquant
					→ l'exécutable n'existe plus à son chemin initial

			⚠️ "Missing Process" + nom suspect = très important
				→ le programme a probablement été exécuté puis supprimé / déplacé

		Étape 3 | Filtrage dans WinPrefetchView
			* Rechercher les noms trouvés en Phase 4 / 4.6 :
				- nom du .exe créé / supprimé dans USN
				- nom du .bat suspect
				- nom utilisé dans un hardlink / rename
				- java.exe / javaw.exe si suspicion de .jar
				- rundll32.exe si suspicion de .dll

			* Mots-clés à surveiller :
				- click
				- autoclicker
				- macro
				- vape
				- raven
				- loader
				- inject
				- bypass
				- ghost
				- clicker

			* Noms système à ne pas ignorer s'ils sont proches de l'heure de jeu :
				- CMD.EXE
				- POWERSHELL.EXE
				- RUNDLL32.EXE
				- REGSVR32.EXE
				- MSHTA.EXE
				- WSCRIPT.EXE / CSCRIPT.EXE
				- JAVA.EXE / JAVAW.EXE

		Étape 4 | Analyse des fichiers chargés
			* Dans WinPrefetchView, sélectionner un Prefetch suspect
			* Observer les fichiers / chemins associés au programme
			* Rechercher :
				- chemins dans AppData / Temp / Downloads / Desktop / OneDrive
				- fichiers .dll chargés depuis un dossier utilisateur
				- chemins vers .jar / .bat / config suspecte
				- fichiers supprimés ou introuvables

			⚠️ Un loader peut avoir un nom banal mais charger des chemins suspects.

		Étape 5 | Export / conservation
			* Dans WinPrefetchView :
				* Sélectionner les lignes suspectes
				* Faire un export texte / CSV si disponible
				* Conserver :
					- nom du Prefetch
					- nom de l'exécutable
					- dernière exécution
					- run count
					- chemin du processus
					- état "missing" si présent

			* Alternative PowerShell :
				Get-ChildItem -Path 'C:\Windows\Prefetch' -Filter *.pf | Select-Object Name, LastWriteTime, Length | Sort-Object LastWriteTime -Descending > "$env:USERPROFILE\Desktop\prefetch_recent.txt"

		Interprétation :
			* .exe suspect présent dans Prefetch + absent du disque
				→ exécutable lancé puis supprimé
			* .exe suspect exécuté juste avant / pendant Minecraft
				→ corrélation forte
			* CMD / PowerShell / BAT proche d'une suppression USN
				→ possible script de nettoyage
			* RUNDLL32.EXE proche d'une .dll suspecte
				→ vérifier Phase DLL / System32 / USN
			* JAVA/JAVAW.EXE proche d'une création/suppression .jar
				→ vérifier Phase 2 et Phase 4.6
			* Dossier Prefetch vide ou nettoyé récemment
				→ red flag MAJEUR, passer à SRUM / Windows.db / BAM / Amcache
				→ Voir ANNEXE 5.a

		⚠️ Remarques importantes :
			* La suppression d’un fichier Prefetch est un red flag MAJEUR, car Windows perd la trace des exécutions
			* Last Run Time est plus utile que la simple date du fichier .pf quand WinPrefetchView l'affiche
			* Les Prefetch ne contiennent pas toujours le chemin exact du fichier supprimé
			* Un Prefetch seul = preuve d'exécution, PAS preuve de cheat
			* La preuve devient forte avec :
				→ USN : création / suppression / renommage
				→ Prefetch : exécution
				→ SRUM : activité pendant Minecraft
				→ Mémoire : traces ou injection active

	Phase 6 – AMCACHE (historique des exécutables)
		Objectif :
			* Retrouver des exécutables supprimés
			* Identifier des fichiers .exe / .dll présents ou inventoriés par Windows
			* Retrouver le chemin d'un loader supprimé
			* Obtenir un hash SHA1 exploitable pour comparaison
			* Corréler un nom trouvé en USN / Prefetch avec un chemin réel
			* Repérer des exécutables dans AppData / Temp / Downloads / Desktop / OneDrive

		Rappel important :
			* Amcache.hve est une ruche Windows située ici :
				C:\Windows\AppCompat\Programs\Amcache.hve
			* Amcache garde des informations sur des fichiers vus / inventoriés par Windows
			* Amcache peut contenir :
				- chemin du fichier
				- nom du programme
				- éditeur / publisher
				- SHA1
				- taille
				- timestamps
			* Amcache ne prouve PAS toujours une exécution directe
			* Amcache devient fort quand il est corrélé avec :
				→ Prefetch : preuve d'exécution
				→ USN : création / suppression / renommage
				→ BAM : exécution par utilisateur
				→ SRUM : activité pendant Minecraft

		Outils requis :
			* AmcacheParser					→ Voir Tools Bible
			* Timeline Explorer				→ Voir Tools Bible
			* Registry Explorer (optionnel)	→ Voir Tools Bible
			⚠️ Si outil bloqué / CSV vide / accès refusé		→ Voir ANNEXE 6.a

		Étape 1 | Préparer le dossier de travail
			* Windows + X -> Terminal / PowerShell (Administrateur)
			* Créer un dossier sur le Bureau :
				$Desktop = [Environment]::GetFolderPath("Desktop")
				New-Item -ItemType Directory -Path "$Desktop\VerifAmcache" -Force

			* Copier la ruche Amcache :
				Copy-Item "C:\Windows\AppCompat\Programs\Amcache.hve" -Destination "$Desktop\VerifAmcache\Amcache.hve" -Force

			* Copier aussi les logs si présents :
				Get-ChildItem "C:\Windows\AppCompat\Programs" -Filter "Amcache.hve.LOG*" -ErrorAction SilentlyContinue | Copy-Item -Destination "$Desktop\VerifAmcache\" -Force

			⚠️ Si la copie échoue :
				→ lancer PowerShell en Administrateur
				→ utiliser AmcacheParser directement sur le fichier système
				→ Voir ANNEXE 6.a

		Étape 2 | Parser avec AmcacheParser (méthode recommandée)
			* Placer AmcacheParser.exe dans un dossier dédié
			* Ouvrir Terminal / PowerShell Administrateur dans ce dossier
			* Taper la commande :
				.\AmcacheParser.exe -f "$env:USERPROFILE\Desktop\VerifAmcache\Amcache.hve" --csv "$env:USERPROFILE\Desktop\VerifAmcache"

			* Résultat attendu :
				→ plusieurs fichiers CSV générés dans :
					Bureau\VerifAmcache

			* Fichiers CSV à analyser en priorité :
				- Amcache_UnassociatedFileEntries.csv
				- Amcache_InventoryApplicationFile.csv
				- Amcache_InventoryApplication.csv
				- Amcache_InventoryDriverBinary.csv

			⚠️ Les noms exacts peuvent varier selon la version de Windows / AmcacheParser.

		Étape 3 | Analyse dans Timeline Explorer
			* Ouvrir Timeline Explorer
			* Charger les CSV générés par AmcacheParser
			* Trier / filtrer sur les colonnes utiles :
				- Path / FullPath / LowerCaseLongPath
				- Name / FileName
				- ProgramName
				- Publisher
				- SHA1
				- Size
				- LinkDate
				- LastModifiedTime

			* Filtrer les chemins utilisateur :
				Contains([Path], 'C:\Users\')
				OR Contains([FullPath], 'C:\Users\')
				OR Contains([LowerCaseLongPath], 'c:\users\')

			* Rechercher les emplacements suspects :
				- Downloads
				- Desktop
				- AppData\Local\Temp
				- AppData\Roaming
				- OneDrive
				- dossier au nom aléatoire

			* Rechercher les extensions / noms :
				- .exe
				- .dll
				- clicker
				- autoclicker
				- macro
				- loader
				- inject
				- bypass
				- vape
				- raven
				- ghost

		Étape 4 | Analyse des résultats
			* Pour chaque entrée suspecte, noter :
				- nom du fichier
				- chemin complet
				- SHA1
				- taille
				- publisher / éditeur
				- timestamps disponibles

			* Indicateurs forts :
				* Exécutable dans Downloads / Desktop / Temp
					→ loader portable possible
				* Publisher vide / inconnu
					→ suspect si chemin utilisateur
				* SHA1 présent mais fichier absent du disque
					→ fichier probablement supprimé
				* Nom aléatoire + chemin utilisateur
					→ suspect
				* DLL dans AppData / Temp
					→ vérifier injection / rundll32 / regsvr32

			* Vérifier si le fichier existe encore :
				Test-Path "C:\chemin\vers\fichier.exe"

			* Si le fichier existe encore, calculer son hash :
				Get-FileHash -Path "C:\chemin\vers\fichier.exe" -Algorithm SHA256

			* Comparer le hash / nom :
				- VirusTotal
				- MalwareBazaar
				- recherche Discord / ticket interne si connu

		Étape 5 | Méthode Registry Explorer (optionnelle / lecture GUI)
			Quand utiliser :
				* AmcacheParser ne fonctionne pas
				* besoin de visualiser la ruche directement
				* besoin de rechercher un nom précis dans Amcache.hve

			1. Ouvrir Registry Explorer en Administrateur
			2. File -> Load Hive
			3. Sélectionner :
				Bureau\VerifAmcache\Amcache.hve
			4. Utiliser la recherche :
				- nom du fichier trouvé en USN
				- nom du fichier Prefetch
				- mot-clé suspect
				- hash / fragment de chemin
			5. Explorer les zones :
				- InventoryApplicationFile
				- InventoryApplication
				- InventoryDriverBinary
				- File / UnassociatedFileEntries selon version

			⚠️ Registry Explorer est utile pour confirmer une entrée, mais AmcacheParser + CSV reste plus pratique pour filtrer massivement.

		Étape 6 | Corrélation obligatoire
			* Si Amcache trouve un exécutable suspect :
				1. Chercher son nom dans Phase 5 - Prefetch
				2. Chercher son nom / chemin dans Phase 4.6 - USN
				3. Chercher son activité dans Phase 10 - SRUM
				4. Chercher son utilisateur via Phase 7 - BAM
				5. Chercher dans Windows.db si besoin

			* Exemple de corrélation forte :
				18:27 → USN : loader.exe créé dans Downloads
				18:28 → Prefetch : LOADER.EXE exécuté
				18:29 → SRUM : loader.exe actif pendant javaw.exe
				18:31 → USN : loader.exe supprimé
				Amcache → chemin + SHA1 du loader

			→ Ici Amcache permet d'identifier précisément le fichier et son hash.

		Interprétation :
			* Amcache seul :
				→ indice de présence / inventaire du fichier
			* Amcache + Prefetch :
				→ fichier identifié + exécution
			* Amcache + Prefetch + SRUM :
				→ fichier identifié + exécution + activité pendant Minecraft
			* Amcache + USN suppression :
				→ fichier identifié puis probablement nettoyé

		⚠️ Conclusion :
			* Amcache est excellent pour retrouver le QUOI
			* Prefetch / BAM / SRUM prouvent mieux le QUAND et l'EXÉCUTION
			* Ne jamais conclure sur Amcache seul

	Phase 7 – BAM (exécution par utilisateur)
		Objectif :
			* Associer un exe à un utilisateur précis
			* Retrouver le dernier lancement connu d'un programme
			* Identifier quel compte Windows a lancé l'exécutable
			* Confirmer un exécutable trouvé en USN / Amcache / Prefetch
			* Obtenir une preuve utile quand Prefetch est vide ou nettoyé
			* Corréler l'activité avec l'heure de lancement de Minecraft

		Rappel important :
			* BAM = Background Activity Moderator
			* La clé BAM associe :
				→ un SID utilisateur
				→ un chemin de programme
				→ un timestamp de dernière activité / exécution
			* Emplacement principal :
				HKLM\SYSTEM\CurrentControlSet\Services\bam\State\UserSettings
			* Ancien emplacement possible selon version Windows :
				HKLM\SYSTEM\CurrentControlSet\Services\bam\UserSettings
			* Emplacement secondaire DAM possible :
				HKLM\SYSTEM\CurrentControlSet\Services\dam\State\UserSettings
			* BAM ne remplace PAS Prefetch, mais complète très bien la timeline

		Limites :
			* BAM ne liste pas forcément tous les programmes
			* BAM peut être incomplet selon version Windows / nettoyage / politique système
			* Le chemin est souvent au format :
				\Device\HarddiskVolumeX\Users\...\Downloads\loader.exe
			* BAM seul = indice d'exécution par utilisateur, à corréler

		Outils requis :
			* PowerShell / Regedit natif Windows
			* Registry Explorer					→ Voir Tools Bible
			* RECmd (optionnel / avancé)			→ Voir Tools Bible
			* Timeline Explorer pour lire les CSV	→ Voir Tools Bible
			⚠️ Si accès refusé / SID illisible / clé absente	→ Voir ANNEXE 7.a

		Étape 1 | Créer le dossier de travail
			* Windows + X -> Terminal / PowerShell (Administrateur)
			* Taper :
				$Desktop = [Environment]::GetFolderPath("Desktop")
				New-Item -ItemType Directory -Path "$Desktop\VerifBAM" -Force

		Étape 2 | Exporter les clés BAM / DAM
			* Exporter BAM :
				reg export "HKLM\SYSTEM\CurrentControlSet\Services\bam\State\UserSettings" "$env:USERPROFILE\Desktop\VerifBAM\bam_UserSettings.reg" /y

			* Exporter ancien chemin BAM si présent :
				reg export "HKLM\SYSTEM\CurrentControlSet\Services\bam\UserSettings" "$env:USERPROFILE\Desktop\VerifBAM\bam_legacy_UserSettings.reg" /y

			* Exporter DAM si présent :
				reg export "HKLM\SYSTEM\CurrentControlSet\Services\dam\State\UserSettings" "$env:USERPROFILE\Desktop\VerifBAM\dam_UserSettings.reg" /y

			⚠️ Si l'ancien chemin BAM n'existe pas, ce n'est pas grave.
			⚠️ Si DAM n'existe pas, ce n'est pas grave.
			⚠️ Si BAM n'existe pas ou accès refusé	→ Voir ANNEXE 7.a

		Étape 3 | Parser BAM avec PowerShell (méthode rapide)
			* Taper la commande :
				$base='HKLM:\SYSTEM\CurrentControlSet\Services\bam\State\UserSettings'; $rows=foreach($k in Get-ChildItem $base -ErrorAction SilentlyContinue){$sid=$k.PSChildName; try{$user=([Security.Principal.SecurityIdentifier]$sid).Translate([Security.Principal.NTAccount]).Value}catch{$user=$sid}; $props=(Get-ItemProperty $k.PSPath).PSObject.Properties | Where-Object { $_.Name -like '\Device\*' -and $_.Value -is [byte[]] -and $_.Value.Length -ge 8 }; foreach($p in $props){[PSCustomObject]@{User=$user;SID=$sid;Path=$p.Name;LastRunLocal=[DateTime]::FromFileTimeUtc([BitConverter]::ToInt64($p.Value,0)).ToLocalTime()}}}; $rows | Sort-Object LastRunLocal -Descending | Export-Csv "$env:USERPROFILE\Desktop\VerifBAM\bam_parsed.csv" -NoTypeInformation -Encoding UTF8; $rows | Sort-Object LastRunLocal -Descending | Format-Table -Auto

			* Résultat :
				→ fichier créé :
					Bureau\VerifBAM\bam_parsed.csv

			* Pour DAM si la clé existe :
				$base='HKLM:\SYSTEM\CurrentControlSet\Services\dam\State\UserSettings'; $rows=foreach($k in Get-ChildItem $base -ErrorAction SilentlyContinue){$sid=$k.PSChildName; try{$user=([Security.Principal.SecurityIdentifier]$sid).Translate([Security.Principal.NTAccount]).Value}catch{$user=$sid}; $props=(Get-ItemProperty $k.PSPath).PSObject.Properties | Where-Object { $_.Name -like '\Device\*' -and $_.Value -is [byte[]] -and $_.Value.Length -ge 8 }; foreach($p in $props){[PSCustomObject]@{User=$user;SID=$sid;Path=$p.Name;LastRunLocal=[DateTime]::FromFileTimeUtc([BitConverter]::ToInt64($p.Value,0)).ToLocalTime()}}}; $rows | Sort-Object LastRunLocal -Descending | Export-Csv "$env:USERPROFILE\Desktop\VerifBAM\dam_parsed.csv" -NoTypeInformation -Encoding UTF8; $rows | Sort-Object LastRunLocal -Descending | Format-Table -Auto

			* Pour ancien BAM si la clé existe :
				$base='HKLM:\SYSTEM\CurrentControlSet\Services\bam\UserSettings'; $rows=foreach($k in Get-ChildItem $base -ErrorAction SilentlyContinue){$sid=$k.PSChildName; try{$user=([Security.Principal.SecurityIdentifier]$sid).Translate([Security.Principal.NTAccount]).Value}catch{$user=$sid}; $props=(Get-ItemProperty $k.PSPath).PSObject.Properties | Where-Object { $_.Name -like '\Device\*' -and $_.Value -is [byte[]] -and $_.Value.Length -ge 8 }; foreach($p in $props){[PSCustomObject]@{User=$user;SID=$sid;Path=$p.Name;LastRunLocal=[DateTime]::FromFileTimeUtc([BitConverter]::ToInt64($p.Value,0)).ToLocalTime()}}}; $rows | Sort-Object LastRunLocal -Descending | Export-Csv "$env:USERPROFILE\Desktop\VerifBAM\bam_legacy_parsed.csv" -NoTypeInformation -Encoding UTF8; $rows | Sort-Object LastRunLocal -Descending | Format-Table -Auto

			* Colonnes générées :
				* User			→ utilisateur Windows traduit depuis le SID
				* SID			→ identifiant utilisateur brut
				* Path			→ chemin de l'exécutable
				* LastRunLocal	→ timestamp local décodé

			* Affichage optionnel :
				* Remplacer `Format-Table -Auto` par `Out-GridView` si disponible
				* Pratique pour trier / filtrer visuellement les entrées BAM

		Étape 4 | Filtrer les chemins suspects
			* Dans PowerShell :
				Import-Csv "$env:USERPROFILE\Desktop\VerifBAM\bam_parsed.csv" | Where-Object { $_.Path -match 'Users|AppData|Temp|Downloads|Desktop|OneDrive|\.exe|\.bat|\.ps1|\.jar' } | Sort-Object LastRunLocal -Descending | Format-Table -Auto

			* Rechercher en priorité :
				- chemins utilisateur
				- Downloads / Desktop / Temp / AppData / OneDrive
				- noms aléatoires
				- autoclicker / clicker / macro / loader / inject / bypass
				- java.exe / javaw.exe proche d'un .jar suspect
				- rundll32.exe / regsvr32.exe proche d'une DLL suspecte
				- cmd.exe / powershell.exe proche d'une suppression USN

		Étape 5 | Lecture avec Timeline Explorer
			* Ouvrir Timeline Explorer
			* Charger :
				Bureau\VerifBAM\bam_parsed.csv

			* Trier par :
				LastRunLocal décroissant

			* Filtrer :
				- User = utilisateur vérifié
				- Path contient Users / AppData / Temp / Downloads / Desktop / OneDrive
				- Path contient le nom suspect trouvé en Phase 4.6 / Phase 5 / Phase 6

			* Comparer avec :
				- heure de connexion serveur
				- heure de lancement Minecraft
				- heure de dernière exécution Prefetch
				- timestamp SRUM

		Étape 6 | Lecture avec Registry Explorer (méthode GUI)
			Quand utiliser :
				* PowerShell ne fonctionne pas
				* besoin de vérifier visuellement la clé
				* besoin de contrôler un SID précis

			1. Ouvrir Registry Explorer en Administrateur
			2. Charger la ruche SYSTEM :
				C:\Windows\System32\config\SYSTEM
			3. Naviguer vers :
				ControlSet001\Services\bam\State\UserSettings
			4. Ouvrir le SID correspondant à l'utilisateur
			5. Observer les valeurs :
				- noms des valeurs = chemins exécutés
				- données = timestamp FILETIME

			⚠️ CurrentControlSet n'existe pas toujours sous ce nom dans une ruche chargée hors système.
			⚠️ Utiliser ControlSet001 ou ControlSet002 selon la ruche.
			⚠️ Voir ANNEXE 7.a pour identifier le bon ControlSet.

		Étape 7 | Méthode RECmd (avancée / export registre)
			Quand utiliser :
				* besoin d'un export CSV propre
				* analyse hors-ligne d'une ruche SYSTEM copiée
				* PowerShell / Regedit bloqué

			* Copier la ruche SYSTEM :
				Copy-Item "C:\Windows\System32\config\SYSTEM" -Destination "$env:USERPROFILE\Desktop\VerifBAM\SYSTEM" -Force

			* Lancer RECmd depuis son dossier :
				.\RECmd.exe -f "$env:USERPROFILE\Desktop\VerifBAM\SYSTEM" --kn "ControlSet001\Services\bam\State\UserSettings" --csv "$env:USERPROFILE\Desktop\VerifBAM"

			* Si rien ne sort :
				→ essayer ControlSet002
				→ vérifier ANNEXE 7.a

			⚠️ RECmd est surtout utile pour exporter / documenter.
			⚠️ Le parsing PowerShell reste plus direct pour convertir les timestamps BAM rapidement.

		Étape 8 | Interprétation
			* BAM montre un loader suspect lancé par l'utilisateur
				→ preuve forte si corrélée avec Minecraft
			* BAM montre un .exe supprimé ensuite dans USN
				→ exécution + nettoyage probable
			* BAM montre cmd.exe / powershell.exe proche d'une suppression
				→ possible script de nettoyage
			* BAM montre javaw.exe proche d'une activité .jar
				→ corréler Phase 2 / Phase 4.6 / Phase 5
			* BAM ne montre rien
				→ ne pas conclure à l'absence de cheat

		Exemple de corrélation forte :
			18:22 → BAM : \Device\HarddiskVolumeX\Users\...\Downloads\loader.exe
			18:23 → Prefetch : LOADER.EXE exécuté
			18:24 → Minecraft / javaw.exe actif
			18:25 → SRUM : loader.exe et javaw.exe actifs
			18:30 → USN : loader.exe supprimé

			→ BAM permet d'associer l'exécution à l'utilisateur.

		Conclusion :
			* BAM répond surtout à QUI + QUAND
			* Prefetch répond mieux à EXÉCUTÉ
			* Amcache répond mieux à QUOI + HASH
			* SRUM répond mieux à ACTIF PENDANT MINECRAFT
			* BAM seul = indice fort, mais corrélation obligatoire

	Phase 8 – NTUSER.DAT (activité utilisateur)
		Objectif :
			* Voir ce que l’utilisateur a ouvert / tapé
			* Identifier des fichiers ouverts / sélectionnés par l'utilisateur
			* Retrouver des commandes tapées via Windows + R
			* Repérer des programmes lancés manuellement via l'interface Windows
			* Détecter des scripts / archives / loaders manipulés récemment
			* Corréler l'activité utilisateur avec Minecraft, USN, Prefetch, BAM et SRUM

		Rappel important :
			* NTUSER.DAT est la ruche registre propre à un utilisateur
			* Emplacement :
				C:\Users\<Utilisateur>\NTUSER.DAT
			* Fichier complémentaire utile :
				C:\Users\<Utilisateur>\AppData\Local\Microsoft\Windows\UsrClass.dat
			* NTUSER.DAT montre surtout l'activité utilisateur :
				→ fichiers récents
				→ commandes Windows + R
				→ programmes lancés via GUI
				→ dialogues ouvrir/enregistrer
				→ chemins tapés dans l'explorateur

		Limites :
			* NTUSER.DAT ne prouve pas forcément l'exécution complète d'un programme
			* Certaines valeurs sont encodées / binaires
			* Une trace RecentDocs peut seulement indiquer ouverture / sélection
			* UserAssist indique des lancements GUI, pas tous les lancements possibles
			* NTUSER devient fort quand il est corrélé avec Prefetch / BAM / SRUM

		Outils requis :
			* Registry Explorer				→ Voir Tools Bible
			* RECmd							→ Voir Tools Bible
			* UserAssistView					→ Voir Tools Bible
			* RecentFilesView / LastActivityView	→ Voir Tools Bible
			⚠️ Si ruche verrouillée / SID inconnu / valeurs illisibles	→ Voir ANNEXE 8.a

		Étape 1 | Créer le dossier de travail
			* Windows + X -> Terminal / PowerShell (Administrateur)
			* Taper :
				$Desktop = [Environment]::GetFolderPath("Desktop")
				New-Item -ItemType Directory -Path "$Desktop\VerifNTUSER" -Force

		Étape 2 | Identifier l'utilisateur vérifié
			* Utilisateur actuellement connecté :
				whoami
				whoami /user

			* Lister les profils locaux :
				Get-ChildItem 'C:\Users' | Select-Object Name, FullName, LastWriteTime

			* Trouver les chemins profils via registre :
				Get-ItemProperty 'HKLM:\SOFTWARE\Microsoft\Windows NT\CurrentVersion\ProfileList\*' | Select-Object PSChildName, ProfileImagePath

			⚠️ Noter le nom utilisateur et le SID.

		Étape 3 | Copier / sauvegarder NTUSER.DAT
			* Méthode recommandée pour l'utilisateur courant :
				reg save HKCU "$env:USERPROFILE\Desktop\VerifNTUSER\NTUSER_HKCU.hiv" /y

			* Copier les fichiers utilisateur si possible :
				Copy-Item "$env:USERPROFILE\NTUSER.DAT" -Destination "$env:USERPROFILE\Desktop\VerifNTUSER\NTUSER.DAT" -Force
				Copy-Item "$env:USERPROFILE\NTUSER.DAT.LOG*" -Destination "$env:USERPROFILE\Desktop\VerifNTUSER\" -Force -ErrorAction SilentlyContinue

			* Copier UsrClass.dat si possible :
				Copy-Item "$env:LOCALAPPDATA\Microsoft\Windows\UsrClass.dat" -Destination "$env:USERPROFILE\Desktop\VerifNTUSER\UsrClass.dat" -Force
				Copy-Item "$env:LOCALAPPDATA\Microsoft\Windows\UsrClass.dat.LOG*" -Destination "$env:USERPROFILE\Desktop\VerifNTUSER\" -Force -ErrorAction SilentlyContinue

			⚠️ Si la copie échoue ou fichier verrouillé	→ Voir ANNEXE 8.a

		Étape 4 | Analyse avec Registry Explorer (méthode GUI)
			1. Ouvrir Registry Explorer en Administrateur
			2. File -> Load Hive
			3. Charger :
				Bureau\VerifNTUSER\NTUSER_HKCU.hiv
				ou
				Bureau\VerifNTUSER\NTUSER.DAT
			4. Rechercher les clés prioritaires :
				Software\Microsoft\Windows\CurrentVersion\Explorer\RecentDocs
				Software\Microsoft\Windows\CurrentVersion\Explorer\RunMRU
				Software\Microsoft\Windows\CurrentVersion\Explorer\ComDlg32\OpenSavePidlMRU
				Software\Microsoft\Windows\CurrentVersion\Explorer\ComDlg32\LastVisitedPidlMRU
				Software\Microsoft\Windows\CurrentVersion\Explorer\UserAssist
				Software\Microsoft\Windows\CurrentVersion\Explorer\TypedPaths

			* Rechercher les mots-clés :
				- .exe
				- .jar
				- .bat
				- .ps1
				- .dll
				- clicker
				- autoclicker
				- macro
				- loader
				- inject
				- bypass
				- vape
				- raven
				- ghost

		Étape 5 | Clés prioritaires à interpréter
			* RecentDocs
				→ fichiers récemment ouverts / manipulés
				→ très utile pour archives, scripts, exécutables sélectionnés

			* RunMRU
				→ commandes tapées dans Windows + R
				→ rechercher cmd, powershell, chemins directs, scripts, URLs

			* ComDlg32\OpenSavePidlMRU
				→ fichiers vus dans une fenêtre Ouvrir / Enregistrer
				→ utile pour savoir si un loader / script a été sélectionné

			* ComDlg32\LastVisitedPidlMRU
				→ dernières applications ayant ouvert des boîtes de dialogue fichiers
				→ utile pour lier un programme à un fichier choisi

			* UserAssist
				→ programmes lancés via interface graphique
				→ contient souvent un compteur et une date de dernière exécution

			* TypedPaths
				→ chemins tapés dans l'explorateur
				→ utile pour Downloads, AppData, Temp, dossiers cachés

		Étape 6 | UserAssistView (méthode rapide)
			* Télécharger / ouvrir UserAssistView
				https://www.nirsoft.net/utils/userassistview.zip
			* Lancer en Administrateur
			* Trier par date de dernière exécution / modified time
			* Rechercher :
				- exécutables lancés proche de Minecraft
				- chemins utilisateur suspects
				- noms aléatoires
				- programmes non signés / inconnus

			⚠️ UserAssistView est pratique pour la lecture rapide, mais Registry Explorer / RECmd permettent une meilleure conservation.

		Étape 7 | RECmd (export avancé)
			* Se placer dans le dossier RECmd
			* Exporter une clé précise :
				.\RECmd.exe -f "$env:USERPROFILE\Desktop\VerifNTUSER\NTUSER_HKCU.hiv" --kn "Software\Microsoft\Windows\CurrentVersion\Explorer\RunMRU" --csv "$env:USERPROFILE\Desktop\VerifNTUSER"

			* Autres clés utiles :
				Software\Microsoft\Windows\CurrentVersion\Explorer\RecentDocs
				Software\Microsoft\Windows\CurrentVersion\Explorer\ComDlg32\OpenSavePidlMRU
				Software\Microsoft\Windows\CurrentVersion\Explorer\ComDlg32\LastVisitedPidlMRU
				Software\Microsoft\Windows\CurrentVersion\Explorer\UserAssist
				Software\Microsoft\Windows\CurrentVersion\Explorer\TypedPaths

			⚠️ Si RECmd ne retourne rien	→ Voir ANNEXE 8.a

		Étape 8 | Interprétation
			* RunMRU contient powershell / cmd + script suspect
				→ forte suspicion de lancement manuel
			* UserAssist contient un loader proche de Minecraft
				→ lancement GUI probable par l'utilisateur
			* RecentDocs contient archive / script / exe suspect
				→ fichier manipulé ou ouvert récemment
			* ComDlg32 contient un .jar / .exe / .dll dans Downloads
				→ fichier sélectionné via une application
			* TypedPaths contient AppData / Temp / dossier caché
				→ utilisateur a navigué volontairement vers ce dossier

		Conclusion :
			* NTUSER répond surtout à l'activité UTILISATEUR
			* NTUSER ne remplace pas Prefetch / BAM / SRUM
			* NTUSER devient très fort si :
				→ UserAssist montre le lancement
				→ Prefetch prouve l'exécution
				→ BAM associe au même utilisateur
				→ SRUM place l'activité pendant Minecraft

	Phase 9 – Windows.db (timeline utilisateur)
		Objectif :
			* Construire une timeline d’activité
			* Identifier des fichiers visibles / indexés par Windows Search
			* Repérer des exécutables, scripts, archives ou URLs suspectes
			* Retrouver des traces de fichiers supprimés ou renommés
			* Regrouper des fragments de données autour d'un même objet
			* Orienter les contrôles Prefetch, SRUM, BAM, Amcache et mémoire

		Rappel important :
			* Windows.db est la base Windows Search
			* Emplacement courant :
				C:\ProgramData\Microsoft\Search\Data\Applications\Windows\Windows.db
			* Windows.db peut contenir des chemins, noms, métadonnées, URLs et timestamps
			* Les informations sont souvent fragmentées :
				→ plusieurs lignes pour un même objet
				→ regroupement par Work Id nécessaire
			* Windows.db seul = indice, pas preuve finale

		Outils requis :
			* Windows Database Parser			→ Voir Tools Bible
			* Timeline Explorer					→ Voir Tools Bible
			* PowerShell / esentutl natif Windows
			⚠️ Si Windows.db est verrouillé / parser bloqué / CSV énorme	→ Voir ANNEXE 9.a

		Étape 1 | Créer le dossier de travail
			* Windows + X -> Terminal / PowerShell (Administrateur)

			* Sélectionner :
				Bureau\VerifWindowsDB\Windows.db

				→ conserver la copie source Windows.db

			* Charger le CSV généré

				- C:\Users\
				- C:\Utilisateurs\
				- Downloads
				- Desktop
				- OneDrive
				- AppData
				- Temp

				- .exe
				- .jar
				- .bat
				- .ps1
				- .dll
				- .zip
				- .rar
				- .7z
				- .ahk
				- .vbs

				- clicker

				- raven
				- ghost
				- github
				- pastebin

			* Trier / grouper par Work Id
			* Pour chaque groupe suspect :
				→ noter toutes les valeurs associées
				→ reconstruire le chemin complet
				→ relever timestamps / URLs / noms
				→ vérifier si le fichier existe encore

			* Vérifier présence disque :
				Test-Path "C:\chemin\vers\fichier.exe"

		Étape 8 | Corrélation
			* Si Windows.db trouve un .exe suspect :
				→ Phase 5 - Prefetch
				→ Phase 6 - Amcache
				→ Phase 7 - BAM
				→ Phase 10 - SRUM
				→ Phase 4.6 - USN

			* Si Windows.db trouve un .jar suspect :
				→ Phase 2 - Java / JNativeHook
				→ Phase 4.6 - activité .jar
				→ Phase 5 - JAVA/JAVAW Prefetch
				→ Phase 10 - SRUM

			* Si Windows.db trouve une URL :
				→ vérifier DNS / navigateur / téléchargements
				→ vérifier fichiers créés après l'heure de l'URL

		Indicateurs :
			* fichier dans Downloads / Desktop / Temp / OneDrive
			* nom aléatoire ou banal
			* .exe / .jar / .bat / .ps1 proche de l'heure de jeu
			* URL GitHub Raw / Pastebin / Discord
			* fichier présent dans Windows.db mais absent du disque
			* groupe Work Id contenant ancien chemin + nouveau nom

		Conclusion :
			* Windows.db indique QUOI chercher
			* Windows.db peut aider à retrouver un fichier supprimé
			* Windows.db ne prouve PAS l'injection ou le cheat seul
			* La preuve vient de la corrélation avec Prefetch / BAM / SRUM / mémoire

	Phase 10 – SRUM (corrélation temporelle)
		Objectif :
			* Vérifier si un loader / cheat a été actif EN MÊME TEMPS que Minecraft
			* Corréler un exécutable suspect avec javaw.exe
			* Voir l'activité CPU / réseau d'un programme
			* Confirmer une activité pendant la session de jeu
			* Renforcer une preuve déjà trouvée via Prefetch, BAM, Amcache ou Windows.db

		Rappel important :
			* SRUM = System Resource Usage Monitor
			* Fichier principal :
				C:\Windows\System32\sru\SRUDB.dat
			* SRUM ne prouve PAS l'injection
			* SRUM prouve surtout :
				→ QUAND un programme a consommé des ressources
				→ SI deux programmes étaient actifs dans la même période
				→ SI un programme a eu une activité réseau

		Limites :
			* SRUM ne montre pas forcément tous les processus
			* Un CpuTime faible ne veut pas dire "pas suspect"
			* Un loader peut être actif quelques secondes seulement
			* Les noms peuvent être incomplets selon les exports
			* SRUM est une preuve de corrélation, pas une preuve de cheat seul

		Outils requis :
			* SrumECmd						→ Voir Tools Bible
			* Timeline Explorer				→ Voir Tools Bible
			* PowerShell / esentutl natif Windows
			⚠️ Si SRUDB.dat est verrouillé / CSV vide / timestamps incohérents	→ Voir ANNEXE 10.a

		Étape 1 | Créer le dossier de travail
			* Windows + X -> Terminal / PowerShell (Administrateur)
			* Taper :
				$Desktop = [Environment]::GetFolderPath("Desktop")
				New-Item -ItemType Directory -Path "$Desktop\VerifSRUM" -Force

		Étape 2 | Copier SRUDB.dat
			* Méthode simple :
				Copy-Item "C:\Windows\System32\sru\SRUDB.dat" -Destination "$env:USERPROFILE\Desktop\VerifSRUM\SRUDB.dat" -Force

			* Si le fichier est verrouillé :
				esentutl /y "C:\Windows\System32\sru\SRUDB.dat" /d "$env:USERPROFILE\Desktop\VerifSRUM\SRUDB.dat"

			⚠️ Si la copie échoue	→ Voir ANNEXE 10.a

		Étape 3 | Parser avec SrumECmd
			* Extraire SrumECmd dans un dossier simple :
				C:\Tools\SrumECmd\

			* Se placer dans le dossier :
				cd C:\Tools\SrumECmd

			* Parser la copie :
				.\SrumECmd.exe -f "$env:USERPROFILE\Desktop\VerifSRUM\SRUDB.dat" --csv "$env:USERPROFILE\Desktop\VerifSRUM"

			* Plusieurs CSV sont générés

		Étape 4 | Fichiers CSV prioritaires
			* Ouvrir dans Timeline Explorer :
				- AppResourceUseInfo.csv
				- NetworkUsages.csv

			* Selon la version de l'outil, le nom exact du CSV peut varier.
			* Rechercher les CSV contenant :
				- AppResource
				- Network
				- Usage

		Étape 5 | Filtrer les processus
			* Dans AppResourceUseInfo.csv :
				- filtrer javaw.exe
				- filtrer java.exe
				- filtrer MinecraftLauncher.exe
				- filtrer le nom du loader suspect
				- filtrer le chemin suspect si le nom seul ne suffit pas

			* Colonnes utiles :
				- Timestamp / TimeStart / TimeEnd selon export
				- ExecutableName / AppId
				- UserId / SID
				- CpuTimeMS
				- EnergyUsage
				- BackgroundBytes / ForegroundBytes selon export

			⚠️ L'objectif n'est pas de trouver un gros pic CPU.
			⚠️ L'objectif est de trouver une présence temporelle proche de Minecraft.

		Étape 6 | Corrélation temporelle
			* Comparer :
				- heure de lancement Minecraft
				- heure de connexion serveur
				- heure du fichier suspect dans Prefetch / BAM / Windows.db / USN
				- activité SRUM de javaw.exe
				- activité SRUM du loader suspect

			* Corrélation forte si :
				- javaw.exe est actif
				- le loader suspect est actif
				- les deux apparaissent dans la même fenêtre temporelle
				- écart d'environ 1 à 5 minutes
				- activité pendant ou juste avant la session de jeu

			Exemple :
				18:22 → Windows.db : loader.exe dans Downloads
				18:23 → Prefetch : LOADER.EXE exécuté
				18:24 → SRUM : loader.exe actif
				18:25 → SRUM : javaw.exe actif
				18:30 → USN : loader.exe supprimé

				→ Corrélation temporelle forte

		Étape 7 | Corrélation réseau
			* Dans NetworkUsages.csv :
				- filtrer javaw.exe
				- filtrer le loader suspect
				- vérifier BytesSent / BytesReceived
				- vérifier Timestamp / UserId / AppId

			* Une activité réseau du loader proche de Minecraft peut indiquer :
				- téléchargement de payload
				- authentification licence
				- communication vers serveur distant
				- update / check de cheat

			⚠️ Une activité réseau seule ne prouve pas un cheat.
			⚠️ Elle devient forte si elle est proche de Prefetch / BAM / Windows.db.

		Interprétation :
			* javaw.exe + loader actif en même temps
				→ preuve de corrélation forte
			* loader actif juste avant Minecraft
				→ lancement préparatoire possible
			* loader actif après fermeture Minecraft
				→ vérifier suppression / nettoyage / update
			* uniquement javaw.exe visible
				→ normal, continuer avec autres artefacts
			* SRUM vide ou absent
				→ ne pas conclure, passer à Prefetch / BAM / Windows.db / MFT

		Preuve forte si :
			* Exécutable suspect identifié
			* Même période que Minecraft
			* Présence dans AppResourceUseInfo ou NetworkUsages
			* Corrélation avec au moins un autre artefact :
				→ Prefetch
				→ BAM
				→ Amcache
				→ Windows.db
				→ USN

		Conclusion :
			* SRUM répond surtout à QUAND
			* SRUM ne prouve pas l'injection
			* SRUM est très fort pour lier un loader à une session Minecraft
			* SRUM + Prefetch + BAM + Windows.db = corrélation solide

	Phase 11 – EventLogs (support)
		Objectif :
			* Appuyer la timeline avec les journaux Windows
			* Rechercher crashs, erreurs, élévations et traces PowerShell
			* Vérifier si des logs ont été vidés
			* Identifier services / tâches planifiées / scripts suspects
			* Compléter SRUM, Prefetch, BAM, Windows.db et NTUSER

		Rappel important :
			* Les EventLogs ne prouvent pas toujours l'exécution d'un cheat
			* Ils servent surtout à confirmer le contexte :
				→ ouverture de session
				→ privilèges administrateur
				→ crash d'un loader
				→ PowerShell / script
				→ tâche planifiée
				→ service installé
				→ nettoyage de logs

		Logs prioritaires :
			* Application.evtx
			* System.evtx
			* Security.evtx
			* Windows PowerShell.evtx
			* Microsoft-Windows-PowerShell/Operational
			* Microsoft-Windows-TaskScheduler/Operational
			* Microsoft-Windows-DNS-Client/Operational
			* Microsoft-Windows-Sysmon/Operational si Sysmon est installé

		Outils requis :
			* Observateur d'événements Windows
			* PowerShell / wevtutil natif Windows
			* EvtxECmd						→ Voir Tools Bible
			* Timeline Explorer				→ Voir Tools Bible
			* PsLogList						→ Voir Tools Bible
			⚠️ Si accès refusé / logs vides / 4688 absent	→ Voir ANNEXE 11.a

		Étape 1 | Créer le dossier de travail
			* Windows + X -> Terminal / PowerShell (Administrateur)
			* Taper :
				$Desktop = [Environment]::GetFolderPath("Desktop")
				$OutDir = Join-Path $Desktop "VerifEventLogs"
				New-Item -ItemType Directory -Path $OutDir -Force

			⚠️ Les commandes utilisant $OutDir doivent être lancées dans la même fenêtre PowerShell.

		Étape 2 | Exporter les logs principaux
			* Taper :
				wevtutil epl Application (Join-Path $OutDir "Application.evtx") /ow:true
				wevtutil epl System (Join-Path $OutDir "System.evtx") /ow:true
				wevtutil epl Security (Join-Path $OutDir "Security.evtx") /ow:true

			* Exporter les logs PowerShell :
				wevtutil epl "Windows PowerShell" (Join-Path $OutDir "WindowsPowerShell.evtx") /ow:true
				wevtutil epl "Microsoft-Windows-PowerShell/Operational" (Join-Path $OutDir "PowerShell_Operational.evtx") /ow:true

			* Exporter les logs complémentaires :
				wevtutil epl "Microsoft-Windows-TaskScheduler/Operational" (Join-Path $OutDir "TaskScheduler_Operational.evtx") /ow:true
				wevtutil epl "Microsoft-Windows-DNS-Client/Operational" (Join-Path $OutDir "DNSClient_Operational.evtx") /ow:true

			* Si Sysmon est installé :
				wevtutil epl "Microsoft-Windows-Sysmon/Operational" (Join-Path $OutDir "Sysmon_Operational.evtx") /ow:true

			⚠️ Si un log n'existe pas, noter l'erreur et continuer.

		Étape 3 | Analyse rapide PowerShell
			* Rechercher les événements Security prioritaires :
				Get-WinEvent -FilterHashtable @{LogName='Security'; Id=1100,1102,1104,4616,4624,4634,4647,4672,4688,4689,4697,4798,4799,6416,6420; StartTime=(Get-Date).AddDays(-7)} -ErrorAction SilentlyContinue | Select-Object TimeCreated, Id, ProviderName, Message | Out-GridView

			* Rechercher les traces PowerShell :
				Get-WinEvent -LogName 'Microsoft-Windows-PowerShell/Operational' -ErrorAction SilentlyContinue | Where-Object { $_.Id -in 4103,4104 } | Select-Object TimeCreated, Id, Message | Out-GridView

			* Rechercher les services installés :
				Get-WinEvent -FilterHashtable @{LogName='System'; Id=7045; StartTime=(Get-Date).AddDays(-7)} -ErrorAction SilentlyContinue | Select-Object TimeCreated, Id, Message | Out-GridView

			* Rechercher nettoyage / arrêt des logs système :
				Get-WinEvent -FilterHashtable @{LogName='System'; Id=104,6005,6006,7036; StartTime=(Get-Date).AddDays(-7)} -ErrorAction SilentlyContinue | Select-Object TimeCreated, Id, ProviderName, Message | Out-GridView

			* Rechercher suppression USN / indices Applications :
				Get-WinEvent -FilterHashtable @{LogName='Application'; Id=3079; StartTime=(Get-Date).AddDays(-7)} -ErrorAction SilentlyContinue | Select-Object TimeCreated, Id, ProviderName, Message | Out-GridView

			* Rechercher les crashs :
				Get-WinEvent -FilterHashtable @{LogName='Application'; Id=1000,1001,1026; StartTime=(Get-Date).AddDays(-7)} -ErrorAction SilentlyContinue | Select-Object TimeCreated, Id, ProviderName, Message | Out-GridView

		Étape 4 | Parser avec EvtxECmd
			* Extraire EvtxECmd dans un dossier simple :
				C:\Tools\EvtxECmd\

			* Se placer dans le dossier :
				cd C:\Tools\EvtxECmd

			* Parser tous les EVTX exportés :
				New-Item -ItemType Directory -Path (Join-Path $OutDir "CSV") -Force
				.\EvtxECmd.exe -d $OutDir --csv (Join-Path $OutDir "CSV")

			* Ouvrir les CSV avec Timeline Explorer

		Étape 5 | Événements prioritaires à vérifier
			* Security.evtx :
				- 1102 → journal d'audit Security supprimé
				- 1100 → service Windows Event Log arrêté / fermé
				- 1104 → journal Security plein
				- 4616 → heure système modifiée
				- 4624 → ouverture de session réussie
				- 4634 → session fermée / compte déconnecté
				- 4647 → déconnexion initiée par l'utilisateur
				- 4648 → connexion avec identifiants explicites
				- 4672 → privilèges spéciaux attribués à une nouvelle connexion
				- 4688 → nouveau processus créé si audit activé
				- 4689 → processus terminé si audit activé
				- 4697 → service installé sur le système
				- 6416 → nouveau périphérique externe reconnu
				- 6420 → périphérique désactivé
				- 4798 → appartenance aux groupes locaux d'un utilisateur énumérée
				- 4799 → appartenance à un groupe local de sécurité énumérée

			⚠️ Notes Security importantes :
				* 4798 / 4799 ne prouvent pas une modification Regedit.
				* 4798 / 4799 ne prouvent pas une suppression de clichés instantanés.
				* Ces IDs peuvent indiquer une énumération de permissions / groupes, à corréler avec processus et horaire.

			* System.evtx :
				- 104 → journal système supprimé
				- 6005 / 6006 → démarrage / arrêt Event Log
				- 7036 → changement état service
				- 7045 → service installé
				- 1074 → redémarrage / arrêt demandé

			* Application.evtx :
				- 3079 → possible suppression USN Journal / vérifier Provider + Message
				- 1000 → crash application
				- 1001 → Windows Error Reporting
				- 1026 → crash .NET Runtime

			* Applications et services :
				- Microsoft-Windows-PowerShell/Operational	→ 4103 / 4104
				- Microsoft-Windows-TaskScheduler/Operational	→ 106 / 140 / 141 / 200 / 201
				- Microsoft-Windows-DNS-Client/Operational	→ requêtes DNS si log activé
				- Microsoft-Windows-Sysmon/Operational		→ 1 / 3 / 7 / 11 / 13 si Sysmon installé

			⚠️ 3079 dépend du Provider et du Message.
			⚠️ Toujours lire le message complet avant de conclure "journal supprimé".

			* PowerShell :
				- 400 / 403 → PowerShell démarré / arrêté
				- 4103 → module logging
				- 4104 → script block logging

			* TaskScheduler :
				- 106 → tâche créée
				- 140 → tâche mise à jour
				- 200 → tâche lancée
				- 201 → tâche terminée

			* Sysmon si présent :
				- 1 → création processus
				- 3 → connexion réseau
				- 7 → DLL chargée
				- 11 → fichier créé
				- 13 → modification registre

		Étape 6 | Mots-clés à rechercher
			* Rechercher dans Message / CommandLine / Path :
				- javaw.exe
				- java.exe
				- powershell
				- cmd.exe
				- wscript
				- cscript
				- mshta
				- rundll32
				- regsvr32
				- schtasks
				- .jar
				- .exe
				- .bat
				- .ps1
				- .dll
				- loader
				- inject
				- bypass
				- github
				- pastebin
				- discord

		Étape 7 | Corrélation
			* Si EventLogs montrent PowerShell proche de Minecraft :
				→ vérifier Phase 4.6 USN pour .ps1 / .bat / .exe
				→ vérifier Phase 9 Windows.db pour URL / payload
				→ vérifier Phase 10 SRUM pour activité simultanée

			* Si EventLogs montrent un service installé :
				→ vérifier chemin du service
				→ vérifier signature / hash
				→ vérifier Prefetch / Amcache / BAM

			* Si EventLogs montrent un journal vidé :
				→ red flag anti-forensic
				→ vérifier USN / Prefetch / SRUM / MFT

			* Si EventLogs montrent un crash de loader :
				→ corréler nom + heure avec Amcache / Prefetch / Windows.db

		Conclusion :
			* EventLogs servent surtout de support
			* Les logs confirment le contexte et les horaires
			* Le log clearing est un red flag majeur
			* EventLogs + SRUM + Prefetch permettent une timeline solide

	Phase 12 – MFT (preuve ultime)
		Objectif :
			* Retrouver les traces fichiers les plus persistantes
			* Identifier des fichiers supprimés, renommés ou déplacés
			* Retrouver anciens chemins / noms / timestamps
			* Confirmer une activité vue dans USN, Prefetch, Windows.db ou BAM
			* Travailler en dernier recours quand les traces classiques sont faibles

		Rappel important :
			* $MFT = Master File Table du volume NTFS
			* Emplacement logique :
				C:\$MFT
			* La MFT contient les enregistrements fichiers du volume :
				→ noms
				→ chemins / parents
				→ timestamps
				→ état supprimé ou encore actif selon les cas
			* La MFT ne prouve PAS l'exécution
			* Elle prouve surtout qu'un fichier a existé sur le disque

		Limites :
			* Les enregistrements supprimés peuvent être réutilisés
			* Sur SSD, TRIM / nettoyage peut réduire les traces récupérables
			* Les timestamps peuvent être modifiés ou difficiles à interpréter
			* Il faut analyser le bon volume : C:, D:, disque externe, etc.
			* MFT seule = preuve d'existence, pas preuve de cheat

		Outils requis :
			* MFTECmd						→ Voir Tools Bible
			* Timeline Explorer				→ Voir Tools Bible
			* PowerShell Administrateur
			⚠️ Si accès refusé / CSV énorme / timestamps confus	→ Voir ANNEXE 12.a

		Étape 1 | Créer le dossier de travail
			* Windows + X -> Terminal / PowerShell (Administrateur)
			* Taper :
				$Desktop = [Environment]::GetFolderPath("Desktop")
				New-Item -ItemType Directory -Path "$Desktop\VerifMFT" -Force

		Étape 2 | Identifier les volumes à analyser
			* Lister les volumes :
				Get-Volume | Select-Object DriveLetter, FileSystem, HealthStatus, SizeRemaining, Size

			* Priorité :
				- C:\
				- disque où se trouve Minecraft
				- disque Downloads / Desktop / OneDrive
				- disque externe si le joueur l'utilise

			⚠️ La MFT n'existe que sur les volumes NTFS.

		Étape 3 | Parser $MFT avec MFTECmd
			* Extraire MFTECmd dans un dossier simple :
				C:\Tools\MFTECmd\

			* Se placer dans le dossier :
				cd C:\Tools\MFTECmd

			* Parser la MFT du disque C:
				.\MFTECmd.exe -f 'C:\$MFT' --csv "$env:USERPROFILE\Desktop\VerifMFT" --csvf mft_C.csv

			* Pour un autre volume :
				.\MFTECmd.exe -f 'D:\$MFT' --csv "$env:USERPROFILE\Desktop\VerifMFT" --csvf mft_D.csv

			⚠️ Utiliser des apostrophes autour de C:\$MFT dans PowerShell.
			⚠️ Sinon PowerShell peut interpréter $MFT comme une variable.

		Étape 4 | Ouvrir dans Timeline Explorer
			* Ouvrir :
				Bureau\VerifMFT\mft_C.csv

			* Désactiver si besoin :
				- Auto Size Columns
				- Auto Refresh

			* Colonnes utiles selon export :
				- EntryNumber
				- FileName / Name
				- Extension
				- ParentPath / FullPath
				- IsDeleted / InUse
				- Created
				- Modified
				- RecordModified / EntryModified
				- LastAccessed

		Étape 5 | Filtrage anti-cheat
			* Filtrer les chemins :
				- \Users\
				- \Downloads\
				- \Desktop\
				- \OneDrive\
				- \AppData\
				- \Temp\
				- \.minecraft\

			* Filtrer les extensions :
				- .exe
				- .jar
				- .bat
				- .ps1
				- .dll
				- .zip
				- .rar
				- .7z
				- .ahk
				- .vbs

			* Filtrer les mots-clés :
				- clicker
				- autoclicker
				- macro
				- loader
				- inject
				- bypass
				- vape
				- raven
				- ghost
				- temp
				- update

		Étape 6 | Identifier suppression / renommage
			* Rechercher :
				- fichiers avec IsDeleted = True
				- fichiers dans Temp / Downloads supprimés rapidement
				- fichiers renommés avec noms différents dans USN
				- fichiers .exe / .jar créés proche de Minecraft
				- fichiers présents dans MFT mais absents du disque

			* Vérifier si le fichier existe encore :
				Test-Path "C:\chemin\vers\fichier.exe"

			* Si MFT montre un fichier absent :
				→ vérifier USN pour suppression
				→ vérifier Prefetch pour exécution
				→ vérifier Amcache pour hash / éditeur
				→ vérifier SRUM pour activité temporelle

		Étape 7 | Lire les timestamps correctement
			* Created
				→ création du fichier sur le volume
			* Modified
				→ contenu modifié
			* RecordModified / EntryModified
				→ entrée MFT modifiée (renommage, déplacement, attributs)
			* LastAccessed
				→ parfois peu fiable selon configuration Windows

			⚠️ Un RecordModified proche d'une suppression / renommage est très important.
			⚠️ Toujours comparer avec l'heure Minecraft et les autres artefacts.

		Étape 8 | Corrélation
			* Exemple de preuve forte :
				18:20 → MFT : loader.exe créé dans Downloads
				18:21 → Prefetch : LOADER.EXE exécuté
				18:22 → SRUM : loader.exe actif
				18:23 → SRUM : javaw.exe actif
				18:30 → USN : loader.exe supprimé

			* MFT + USN :
				→ existence + suppression / renommage
			* MFT + Prefetch :
				→ existence + exécution
			* MFT + Amcache :
				→ existence + hash / métadonnées
			* MFT + SRUM :
				→ existence + activité pendant Minecraft

		Conclusion :
			* MFT répond surtout à QUOI + OÙ + QUAND
			* MFT ne prouve pas l'exécution seule
			* MFT est très utile si le fichier a été supprimé
			* Utiliser la MFT en dernier recours ou pour confirmer une timeline forte

# Analyse Forensique Avancée – Windows.db (Windows Search)

	Cette méthode est utilisée afin d’identifier des exécutables, loaders, scripts ou outils annexes, même supprimés par le joueur, et de construire une timeline d’activité.
	⚠️ Cette méthode ne constitue PAS une preuve directe de cheat ou d’injection, elle sert à orienter les vérifications avancées.

	Objectif de la méthode Windows.db
		* Détecter des exécutables suspects renommés (Vape, Autoclicker, Loader…)
		* Identifier des fichiers supprimés après utilisation
		* Repérer des scripts PowerShell, URL de payload, outils de bypass
		* Corréler une activité suspecte avec une session Minecraft
		* Orienter les contrôles SRUM, Prefetch et Mémoire

	Outils requis
		* Windows Database Parser (Detect.ac ou équivalent) ou WxTCmd.zip
		* Timeline Explorer
		* Accès au fichier :
			C:\ProgramData\Microsoft\Search\Data\Applications\Windows\Windows.db

		Étape 1 – Extraction de Windows.db
			* Extraire le fichier Windows.db
			* Parser la base en format CSV
			* Ouvrir le CSV dans Timeline Explorer
			* Désactiver :
				- Redimensionnement automatique des colonnes
				- Rafraîchissement automatique
			* Masquer les colonnes inutiles :
				- Line
				- ColumnId

		Étape 2 – Compréhension des données (IMPORTANT)
			* Chaque information est fragmentée en plusieurs lignes
			* Une entité réelle = plusieurs lignes avec le même Work Id

				Work Id  → Objet réel (fichier, script, URL)
				Tag      → Type de donnée
				Value    → Valeur brute

			⚠️ Ne jamais analyser ligne par ligne
			⚠️ Toujours regrouper par Work Id

		Étape 3 – Filtrage initial (réduction massive)
			* Appliquer le filtre suivant sur la colonne Value :

				https://gist.github.com/youpl/50fa94e0aba33dc2e2e7c4ad79ec0652

		Étape 4 – Focus utilisateur (recommandé)
			* Ajouter le filtre suivant :

				AND (
					Contains([Value], 'C:\Users\')
					OR Contains([Value], 'C:\Utilisateurs\')
				)

			Cette étape permet d’ignorer totalement les fichiers système.

		Étape 5 – Regroupement logique
			* Cliquer sur la colonne Work Id
			* Appliquer : Group By → Work Id

			Chaque groupe correspond à :
				- Un exécutable
				- Un loader
				- Un script
				- Une URL associée

		Étape 6 – Corrélation temporelle
			* Convertir les timestamps FILETIME en date lisible
			* Comparer avec :
				- Heure de connexion au serveur
				- Heure de lancement de Minecraft

			⚠️ Un fichier suspect actif pendant la session = signal critique

	Indicateurs de suspicion Windows.db
		* Exécutable au nom aléatoire (ex: 55709891.exe)
		* Fichier situé sur Bureau / OneDrive / Temp
		* Absence d’éditeur ou de version
		* Présence de scripts PowerShell (.ps1)
		* URL GitHub / Pastebin / Raw
		* Activité durant la session de jeu

	→ Windows.db seul = suspicion, PAS condamnation

		Ce que Windows.db ne permet PAS de prouver
			* Injection dans javaw.exe
			* Hooks mémoire
			* Cheats fileless
			* DMA
			* Strings chargées en mémoire

	→ Ces preuves doivent être recherchées via :
		- SRUM (CPU / Réseau)
		- Prefetch
		- Analyse mémoire (System Informer)
		- EventLogs / PowerShell

	Suite logique après Windows.db
	Dans le cas où un fichier suspect est identifié :

		1. Vérifier les Prefetch
		2. Vérifier la corrélation SRUM
		3. Analyser la mémoire de javaw.exe
		4. Vérifier les EventLogs et PowerShell

	Règle d’or
		Windows.db indique QUOI chercher,
		mais jamais QUOI conclure.

 ## Vérification de Corrélation SRUM (CPU / Réseau)

	Cette méthode permet de lier temporellement un loader / cheat à Minecraft (javaw.exe) en analysant l’utilisation des ressources système.
	⚠️ SRUM ne montre PAS l’injection, mais prouve que deux programmes ont été actifs au même moment, ce qui constitue une preuve forensique forte.

	Objectif de la méthode SRUM
		* Vérifier si un loader/cheat a été actif en même temps que Minecraft
		* Corréler un exécutable suspect trouvé via Windows.db
		* Confirmer une activité pendant la session de jeu
		* Fournir une preuve temporelle exploitable en vérification

	Outils requis
		* SrumECmd (Eric Zimmerman)
		* Timeline Explorer
		* Accès au fichier :
			C:\Windows\System32\sru\SRUDB.dat

	Étape 1 – Extraction de la base SRUM
		* Copier le fichier suivant :
			C:\Windows\System32\sru\SRUDB.dat

		* Placer SrumECmd.exe dans un dossier dédié

		* Windows + X → Terminal / PowerShell (Administrateur)

		* Se placer dans le dossier contenant SrumECmd :
			cd C:\Chemin\Vers\SrumECmd

		* Lancer la commande suivante :
			SrumECmd.exe -f C:\Windows\System32\sru\SRUDB.dat --csv

		* Attendre la fin du parsing

		→ Plusieurs fichiers CSV sont générés automatiquement

	Étape 2 – Fichiers à analyser (IMPORTANT)
		Ouvrir UNIQUEMENT les fichiers suivants dans Timeline Explorer :

			* AppResourceUseInfo.csv   → Utilisation CPU
			* NetworkUsages.csv        → Activité réseau

		Les autres fichiers SRUM ne sont pas nécessaires dans un contexte anti-cheat.

	Étape 3 – Filtrage des processus
		Dans AppResourceUseInfo.csv :

			* Filtrer la colonne ExecutableName avec :
				javaw.exe

			* Ajouter le nom de l’exécutable suspect identifié précédemment :
				exemple : 55709891.exe

			* Vérifier que les deux processus apparaissent dans la timeline

	Étape 4 – Colonnes importantes à interpréter
		Dans AppResourceUseInfo.csv :

			* Timestamp        → Heure exacte de l’activité
			* ExecutableName   → Nom du processus
			* UserId           → Utilisateur
			* CpuTimeMS        → Temps CPU consommé
			* EnergyUsage      → Indicateur d’activité globale

		⚠️ Un CpuTime faible est NORMAL pour un loader  
		⚠️ L’important est la présence temporelle, pas l’intensité

	Étape 5 – Vérification de la corrélation temporelle
		Une corrélation est considérée comme VALIDE si :

			* javaw.exe et le loader apparaissent
			* dans la même tranche de temps
			* avec un écart de ± 1 à 5 minutes
			* pendant ou juste avant la session de jeu

		Exemple de corrélation valide
		18:32 → javaw.exe        → activité CPU
		18:33 → 55709891.exe     → activité CPU
		18:35 → javaw.exe        → activité CPU

		➡️ Corrélation temporelle confirmée

		Exemple de corrélation NON valide
		10:12 → 55709891.exe
		18:42 → javaw.exe

		➡️ Aucune corrélation exploitable

	Étape 6 – Corrélation réseau (optionnelle mais très forte)
		Dans NetworkUsages.csv :

			* Filtrer sur :
				javaw.exe
				55709891.exe

			* Vérifier une activité réseau simultanée

		Une activité réseau du loader pendant que javaw.exe est actif
		renforce fortement la preuve.

	Quand la corrélation SRUM est suffisante
		La méthode SRUM est considérée comme suffisante si :

			* Un loader ou exécutable suspect est identifié
			* Une corrélation temporelle claire est présente
			* L’activité se situe pendant la session de jeu

	Dans ce cas, l’analyse mémoire n’est PAS obligatoire.

	Erreurs fréquentes à éviter
		* Chercher un pic CPU élevé
		* Chercher le mot “vape” explicitement
		* Ignorer un CpuTime faible
		* Oublier le fuseau horaire
		* Ne pas comparer avec l’heure de connexion serveur

	Suite logique après SRUM
		Si la corrélation est confirmée :

			→ Preuve suffisante dans la majorité des cas

	Phase 20 – Bypass Hardlink / Taskmgr / Conhost
		Objectif :
			* Détecter une exécution contournée via hardlink temporaire
			* Repérer une DLL suspecte cachée dans C:\Windows\System32
			* Retrouver le faux .exe temporaire même s'il est supprimé avant la vérification
			* Vérifier si Taskmgr / conhost ont été lancés automatiquement via tâche planifiée
			* Corréler la méthode avec Minecraft, Java, SRUM, Prefetch, BAM, USN et MFT

		Principe de la méthode observée :
			* Une commande obfusquée crée un hardlink dans %TEMP%
			* Le nom temporaire ressemble à un exécutable légitime ou banal
			* Le hardlink pointe vers une DLL placée dans System32
			* Le faux .exe est lancé puis supprimé
			* Le lancement peut passer par conhost / taskmgr ouvert en administrateur
			* Une tâche planifiée peut préparer l'environnement au démarrage de session

		⚠️ Règle forensic :
			* NE PAS reconstituer la commande d'exécution
			* NE PAS relancer le hardlink ou la DLL suspecte sur le PC du joueur
			* NE PAS supprimer la DLL suspecte avant d'avoir relevé hash / signature / dates
			* Si un test dynamique est nécessaire, il doit être fait hors machine joueur, en environnement isolé

		Outils requis :
			* JournalTrace					→ méthode prioritaire si disponible
			* System Informer / Process Hacker	→ Voir Tools Bible
			* fsutil USN Journal				→ Voir Tools Bible
			* MFTECmd							→ Voir Tools Bible
			* PECmd / WinPrefetchView			→ Voir Tools Bible
			* Sigcheck / Strings				→ Voir Tools Bible
			* EvtxECmd / Timeline Explorer		→ Voir Tools Bible
			⚠️ Si USN vide / hardlink supprimé / tâche planifiée absente	→ Voir ANNEXE 20.a

		Étape 1 | Analyse mémoire javaw.exe
			* Ouvrir System Informer / Process Hacker en Administrateur
			* Rechercher javaw.exe
			* Propriétés -> Memory
			* Désactiver :
				- Hide free page / Hide free regions
			* Ouvrir Strings
			* Minimum length : 4 ou 5
			* Cocher :
				- Private
				- Image
				- Mapped

			* Rechercher :
				- Vape
				- x-client
				- slinky
				- loader
				- inject
				- clicker

			Interprétation :
				* strings explicites cheat dans javaw.exe
					→ preuve directe forte
				* rien dans javaw.exe
					→ ne pas conclure, passer aux artefacts hardlink

		Étape 2 | JournalTrace (méthode prioritaire) 
			Objectif :
				* Retrouver le hardlink complet en une seule lecture
				* Lier le nom supprimé au fichier réel / courant
				* Identifier un cheat déguisé qui ne laisse presque aucune trace classique

			* Ouvrir JournalTrace en Administrateur |Journal Trace Tools Bible
			* Charger / parser le journal USN du disque C:
			* Filtrer la colonne Reason avec :
				Hard Link
				Hard link change
				Modification de lien physique

			* Rechercher aussi par mots-clés :
				- javaw.security
				- .security
				- .pf
				- BLUE.pf
				- jre\lib\security
				- PrismLauncher
				- Eclipse Adoptium
				- AppData\Roaming
				- Temp

			* Double-cliquer l'entrée suspecte
			* Relever dans "Entry information" :
				- USN
				- Name
				- Date
				- Reason
				- Directory
				- Current file
				- File Identification

			Exemple important :
				* Name :
					javaw.security
				* Reason :
					Hard link change | Close
				* Directory :
					C:\Program Files\Eclipse Adoptium\...\jre\lib\security
				* Current file :
					C:\Users\<Utilisateur>\AppData\Roaming\PrismLauncher\java\jre-legacy\bin\BLUE.pf

			Interprétation :
				* Name = nom du hardlink / alias utilisé
				* Directory = dossier où l'alias hardlink a existé
				* Current file = fichier réel / courant lié à l'entrée
				* File Identification identique = même objet NTFS malgré un nom différent

			⚠️ Si javaw.security est supprimé après selfdestruct :
				→ il peut disparaître du disque
				→ JournalTrace peut encore montrer le lien vers BLUE.pf
				→ BLUE.pf devient le fichier à analyser / corréler

			⚠️ BLUE.pf dans AppData / PrismLauncher / Java n'est PAS un Prefetch Windows.
			⚠️ Ne pas confondre avec C:\Windows\Prefetch\*.pf.

			À conserver comme preuve :
				* capture de la liste JournalTrace filtrée sur Hard Link
				* capture de Entry information
				* USN / Date / Reason
				* Name + Current file + Directory

		Étape 2.b | Fallback fsutil si JournalTrace absent
			* Windows + X -> Terminal / PowerShell Administrateur
			* Créer un dossier de sortie compatible Bureau OneDrive :
				$Desktop = [Environment]::GetFolderPath("Desktop")
				$OutDir = Join-Path $Desktop "VerifHardlinkBypass"
				New-Item -ItemType Directory -Path $OutDir -Force

			⚠️ Les commandes utilisant $OutDir doivent être lancées dans la même fenêtre PowerShell.
			⚠️ Si erreur "Path a la valeur Null", relancer les 3 lignes ci-dessus avant de continuer.

			* Alternative si vous êtes déjà dans le dossier VerifHardlinkBypass :
				$OutDir = (Get-Location).Path

			* Exporter le journal USN :
				fsutil usn readjournal C: csv | Out-File (Join-Path $OutDir "usn_all.csv") -Encoding ASCII

			* Rechercher les indices hardlink :
				Select-String -Path (Join-Path $OutDir "usn_all.csv") -Pattern 'Hard Link','Modification de lien physique','0x00010000','0x80010000','javaw.security','\\.security','BLUE\\.pf','audiodg','audiodeg' | ForEach-Object { $_.Line } | Set-Content (Join-Path $OutDir "usn_hardlink_raw.csv") -Encoding UTF8

			⚠️ fsutil USN n'affiche pas le chemin complet du fichier.
			⚠️ Il affiche surtout Name + ParentRef.
			⚠️ Pour retrouver le chemin complet, préférer JournalTrace ou corréler le Name / ParentRef avec Phase 12 - MFT / MFTECmd.

			* À relever :
				- Reason : Hard Link Change
				- Reason : Close
				- chemin contenant \AppData\Local\Temp\
				- chemin contenant \jre\lib\security\
				- nom .exe temporaire
				- nom .security
				- nom .pf hors C:\Windows\Prefetch
				- heure proche de Minecraft
				- apparition / suppression rapide

			Indicateur fort :
				* .exe dans %TEMP% + Hard Link Change + Close
				* ou javaw.security + Hard Link Change + Current file vers un .pf suspect
				* puis disparition du fichier
				* proche de l'heure Minecraft

		Étape 3 | Chercher les DLL suspectes dans System32
			⚠️ Cette étape concerne surtout le bypass type DLL / System32.
			⚠️ Si JournalTrace montre un Current file en .pf dans AppData / PrismLauncher / Java, passer aussi à l'Étape 3.b.

			* Windows + X -> Terminal / PowerShell Administrateur
			* Exporter les DLL System32 non valides / non Microsoft :
				Get-ChildItem "$env:windir\System32" -Filter *.dll -Force | ForEach-Object { $sig = Get-AuthenticodeSignature $_.FullName; [PSCustomObject]@{ Name=$_.Name; Path=$_.FullName; CreationTime=$_.CreationTime; LastWriteTime=$_.LastWriteTime; Length=$_.Length; Signature=$sig.Status; Publisher=$sig.SignerCertificate.Subject } } | Where-Object { $_.Signature -ne 'Valid' -or $_.Publisher -notmatch 'Microsoft' } | Sort-Object CreationTime -Descending | Export-Csv (Join-Path $OutDir "system32_dll_suspect.csv") -NoTypeInformation -Encoding UTF8

			* Rechercher en priorité :
				- DLL créée récemment
				- DLL non signée
				- DLL avec signature invalide
				- DLL avec nom proche d'un composant Windows mais incorrect
				- DLL créée proche de l'heure Minecraft

			* Vérifier le hash d'une DLL suspecte :
				Get-FileHash "C:\Windows\System32\NomSuspect.dll" -Algorithm SHA256

			* Vérifier la signature :
				Get-AuthenticodeSignature "C:\Windows\System32\NomSuspect.dll" | Format-List

			⚠️ Une DLL non Microsoft dans System32 n'est pas automatiquement un cheat.
			⚠️ Elle devient critique si elle correspond à l'heure USN / hardlink / Minecraft.

		Étape 3.b | Cas javaw.security / fichier .pf déguisé
			* Si JournalTrace affiche :
				- Name = javaw.security
				- Reason = Hard link change | Close
				- Current file = ...\BLUE.pf

			* Vérifier si le Current file existe encore :
				Test-Path "C:\Users\<Utilisateur>\AppData\Roaming\PrismLauncher\java\jre-legacy\bin\BLUE.pf"

			* Si le fichier existe encore :
				Get-Item "C:\chemin\vers\BLUE.pf" | Select-Object FullName, CreationTime, LastWriteTime, Length
				Get-FileHash "C:\chemin\vers\BLUE.pf" -Algorithm SHA256
				Get-AuthenticodeSignature "C:\chemin\vers\BLUE.pf" | Format-List

			* Vérifier si le fichier est réellement un PE malgré l'extension .pf :
				$Path="C:\chemin\vers\BLUE.pf"
				$bytes = Get-Content $Path -Encoding Byte -TotalCount 2 -ErrorAction SilentlyContinue
				if ($bytes[0] -eq 0x4D -and $bytes[1] -eq 0x5A) { "PE/MZ détecté" } else { "Pas de signature MZ" }

			* Rechercher des strings si le fichier existe :
				strings64.exe -a -n 5 "C:\chemin\vers\BLUE.pf" > "$env:USERPROFILE\Desktop\blue_pf_strings.txt"

			* Rechercher dans les artefacts :
				- USN : BLUE.pf / javaw.security
				- MFT : BLUE.pf / javaw.security
				- Windows.db : BLUE.pf
				- Amcache : BLUE.pf si inventorié
				- SRUM : activité Java proche de l'heure JournalTrace

			⚠️ Un fichier .pf hors C:\Windows\Prefetch est suspect si :
				→ il est dans AppData / PrismLauncher / Java
				→ il est lié à javaw.security par hardlink
				→ il apparaît proche de l'heure Minecraft
				→ il est supprimé ou selfdestruct ensuite

		Étape 4 | Chercher le faux fichier dans Prefetch / artefacts d'exécution
			* Vérifier les Prefetch du nom .exe trouvé en USN si le bypass utilise un .exe :
				Get-ChildItem "C:\Windows\Prefetch" -Filter "NOMEXE*.pf" | Select-Object Name, LastWriteTime, Length | Sort-Object LastWriteTime -Descending

			* Exemple de cas à surveiller :
				- faux audiodg.exe dans %TEMP%
				- Prefetch AUDIODG.EXE avec horaire différent du comportement normal
				- plusieurs Prefetch du même nom avec hashes différents
				- fichier BLUE.pf hors dossier Prefetch Windows

			* Ouvrir avec WinPrefetchView / PECmd :
				→ vérifier les chemins chargés
				→ vérifier si une DLL suspecte apparaît
				→ comparer avec l'heure Minecraft

			⚠️ audiodg.exe existe légitimement dans System32.
			⚠️ Le signal suspect vient du chemin %TEMP%, du hardlink et de la corrélation.
			⚠️ BLUE.pf dans AppData n'est pas un fichier Prefetch Windows.

		Étape 5 | Vérifier BAM / SRUM / Amcache
			* BAM :
				→ chercher le faux .exe temporaire
				→ chercher \AppData\Local\Temp\
				→ relier au SID utilisateur

			* SRUM :
				→ chercher le nom .exe temporaire
				→ comparer avec javaw.exe
				→ vérifier activité CPU / réseau proche de Minecraft

			* Amcache :
				→ chercher le nom .exe temporaire
				→ chercher le nom de la DLL suspecte
				→ relever hash / path / timestamp si présent

			⚠️ Si audiodg.exe apparaît dans SRUM, ne pas conclure seul.
			⚠️ audiodg.exe légitime peut être actif normalement.

		Étape 6 | Vérifier la tâche planifiée de préparation
			* Exporter les tâches suspectes :
				Get-ScheduledTask | Where-Object { ($_.Actions.Execute -match 'taskmgr|conhost|cmd|powershell|wt\.exe') -or ($_.Principal.RunLevel -eq 'Highest') -or ($_.Settings.Hidden -eq $true) } | Select-Object TaskName, TaskPath, State, @{Name='Actions';Expression={($_.Actions | ForEach-Object {"$($_.Execute) $($_.Arguments)"}) -join ' | '}}, @{Name='Triggers';Expression={($_.Triggers | Out-String)}} | Out-File (Join-Path $OutDir "tasks_suspect.txt") -Encoding UTF8

			* Vérifier les fichiers de tâches :
				Get-ChildItem "C:\Windows\System32\Tasks" -Recurse -Force | Select-Object FullName, CreationTime, LastWriteTime, Length | Sort-Object LastWriteTime -Descending | Out-File (Join-Path $OutDir "tasks_files_recent.txt") -Encoding UTF8

			* Rechercher :
				- tâche déclenchée à l'ouverture de session
				- action lançant taskmgr.exe
				- action lançant cmd.exe / conhost / powershell
				- RunLevel Highest
				- tâche masquée
				- tâche créée ou modifiée récemment

			* Corréler avec EventLogs TaskScheduler :
				- 106 → tâche créée
				- 140 → tâche modifiée
				- 200 → tâche lancée
				- 201 → tâche terminée

		Étape 7 | Vérifier Taskmgr / conhost / cmd
			* Prefetch :
				- TASKMGR.EXE
				- CONHOST.EXE
				- CMD.EXE
				- WT.EXE

			* SRUM :
				- taskmgr.exe actif au démarrage session
				- conhost.exe / cmd.exe proche du hardlink

			* BAM :
				- taskmgr.exe lancé par l'utilisateur
				- cmd.exe / powershell.exe / wt.exe proche de Minecraft

			⚠️ conhost.exe seul est très fréquent.
			⚠️ Taskmgr au démarrage via tâche planifiée + hardlink %TEMP% = signal beaucoup plus fort.

		Étape 8 | Corrélation finale
			Preuve forte si plusieurs éléments concordent :
				* JournalTrace : Hard link change sur javaw.security / alias suspect
				* JournalTrace : Current file vers BLUE.pf ou fichier .pf hors Prefetch Windows
				* USN : Hard Link Change sur un .exe / .security / .pf suspect
				* USN : Close / suppression rapide du même nom
				* System32 : DLL récente non signée / signature invalide si variante DLL
				* Prefetch : faux .exe exécuté proche de Minecraft si variante .exe
				* BAM : même .exe / chemin suspect lié au SID joueur si présent
				* SRUM : javaw.exe actif proche de l'heure JournalTrace
				* TaskScheduler : tâche au logon lançant taskmgr / console en admin
				* Mémoire : strings cheat dans javaw.exe

			Conclusion :
				* 1 seul artefact = indice
				* 2-3 artefacts cohérents = suspicion forte
				* JournalTrace javaw.security -> BLUE.pf + heure Minecraft = preuve très forte de bypass
				* Hardlink USN + DLL System32 suspecte + Prefetch/BAM/SRUM = preuve très solide pour variante DLL
				* Mémoire javaw.exe positive = preuve directe

	1.d Processus suspect fermé / accès refusé / chemin invisible

			* Impossible de lire la ligne de commande

			* Noter l'heure exacte
			* Noter le nom du processus si visible
			* Noter le PID si visible
			* Prendre une capture d'écran si possible
			* NE PAS redémarrer le PC
			* NE PAS relancer le programme suspect

		Étape 1 | Relancer les outils avec privilèges
			* Ouvrir Terminal / PowerShell en Administrateur
			* Ouvrir Process Explorer / System Informer en Administrateur
			* Activer :

			* Lister les processus restants :

			* Chercher les processus lancés depuis l'utilisateur :
				Get-CimInstance Win32_Process | Where-Object { $_.ExecutablePath -match '\\Users\\|\\AppData\\|\\Temp\\|\\Downloads\\|\\Desktop\\|\\OneDrive' } | Select-Object ProcessId, ParentProcessId, Name, ExecutablePath, CommandLine

			* Chercher les lanceurs restants :
				Get-CimInstance Win32_Process | Where-Object { $_.Name -match 'java|cmd|powershell|wscript|cscript|mshta|rundll32|regsvr32|python|node|autohotkey' } | Select-Object ProcessId, ParentProcessId, Name, ExecutablePath, CommandLine

		Étape 3 | Si le processus a disparu
			* Passer immédiatement à :
				- Phase 4 / 4.6 - USN Journal
				- Phase 5 - Prefetch

			* Rechercher :
				- création puis suppression du fichier
				- Prefetch correspondant au nom du processus
				- activité CPU / réseau pendant Minecraft
				- entrée BAM liée à l'utilisateur

		Interprétation :
			* Processus qui disparaît pendant la vérification
				→ comportement suspect si corrélé avec USN / Prefetch
			* Accès refusé sur un processus non système

			* Refus du joueur de laisser analyser le processus
				→ noter le refus et corréler avec les artefacts disque

		⚠️ Un processus fermé n'efface pas tout :
			→ USN peut montrer création / suppression
			→ Prefetch peut prouver l'exécution
			→ SRUM peut montrer l'activité pendant Minecraft
			→ BAM peut associer l'exécution à l'utilisateur

	2.a Java / .jar introuvable / ligne de commande masquée
		Problème :
			* Aucun java.exe / javaw.exe visible
			* La ligne de commande Java est vide
			* Plusieurs javaw.exe sont visibles en même temps
			* Le chemin Java semble légitime mais l'activité reste suspecte
			* JNativeHook est présent dans Temp mais aucun .jar n'est visible

		Règle importante :
			* NE PAS relancer un .jar suspect pour "voir ce qu'il fait"
			* NE PAS fermer Minecraft avant d'avoir noté les horaires
			* Un Java légitime peut appartenir à Minecraft ou à un launcher alternatif
			* Un .jar supprimé peut laisser des traces dans USN / Windows.db même s'il n'est plus visible

		Étape 1 | Relancer la collecte en Administrateur
			* Windows + X -> Terminal / PowerShell (Administrateur)
			* Taper :
				Get-CimInstance Win32_Process | Where-Object { $_.Name -match '^javaw?\.exe$' } | Select-Object ProcessId, ParentProcessId, Name, ExecutablePath, CommandLine, CreationDate

			* Si la ligne de commande reste vide :
				→ ouvrir Process Explorer / System Informer en Administrateur
				→ activer Show Details for All Processes
				→ vérifier les propriétés du processus Java

		Étape 2 | Identifier le parent Java
			* Relever le ParentProcessId
			* Chercher le processus parent :
				$pidParent = 1234
				Get-CimInstance Win32_Process | Where-Object { $_.ProcessId -eq $pidParent } | Select-Object ProcessId, Name, ExecutablePath, CommandLine
			* Remplacer 1234 par le ParentProcessId relevé

			* Interprétation :
				* Parent = Minecraft Launcher / launcher connu
					→ cohérent, mais continuer si horaires suspects
				* Parent = cmd.exe / powershell.exe / fichier inconnu
					→ suspect, vérifier ligne de commande et Prefetch
				* Parent disparu
					→ corréler avec USN / Prefetch / SRUM

		Étape 3 | Cas des launchers multiples
			* Plusieurs launchers peuvent utiliser des Java différents :
				- Minecraft Launcher
				- Lunar Client
				- Badlion
				- Feather
				- Labymod
				- Modrinth
				- Prism Launcher
				- MultiMC

			⚠️ Ne pas conclure "cheat" uniquement parce que Java n'est pas dans `.minecraft`.
			⚠️ Conclure seulement si le chemin / commandline / horaire / artefacts concordent.

		Étape 4 | Si aucun Java n'est actif
			* Chercher les traces indirectes :
				- Phase 4.6 : activité des .jar
				- Phase 5 : Prefetch JAVA.EXE / JAVAW.EXE
				- Phase 7 : BAM
				- Phase 9 : Windows.db
				- Phase 10 : SRUM

			* Rechercher les .jar récents :
				Get-ChildItem "$env:USERPROFILE\Downloads","$env:USERPROFILE\Desktop",$env:TEMP,$env:APPDATA,$env:LOCALAPPDATA -Recurse -Force -Filter *.jar -ErrorAction SilentlyContinue | Select-Object FullName, LastWriteTime, Length | Sort-Object LastWriteTime -Descending

			* Rechercher JNativeHook :
				Get-ChildItem $env:TEMP -Recurse -Force -ErrorAction SilentlyContinue | Where-Object { $_.FullName -match 'JNativeHook|jnativehook|jna' } | Select-Object FullName, LastWriteTime, Length | Sort-Object LastWriteTime -Descending

		Étape 5 | Si un fichier d'arguments Java est visible
			* Certaines lignes de commande Java utilisent un fichier :
				@C:\chemin\vers\arguments.txt
				@C:\chemin\vers\args

			* Si le fichier existe encore :
				Get-Content "C:\chemin\vers\arguments.txt"

			* Rechercher dedans :
				- .jar
				- -cp
				- -classpath
				- chemins AppData / Temp / Downloads / Desktop

		Interprétation :
			* JNativeHook récent + Java actif + Minecraft ouvert
				→ suspicion forte
			* .jar supprimé dans USN + JAVA/JAVAW dans Prefetch
				→ exécution probable à corréler avec SRUM
			* Ligne de commande Java contenant -jar vers un chemin utilisateur
				→ preuve forte d'exécution d'un .jar externe
			* Aucun Java actif mais traces USN / Prefetch / SRUM cohérentes
				→ le .jar a pu être lancé puis fermé

		⚠️ Conclusion :
			* Java actif seul = normal pour Minecraft
			* JNativeHook seul = indice
			* -jar + chemin suspect + horaire Minecraft = preuve forte
			* Corrélation temporelle = élément décisif

	4.a fsutil / USN Journal impossible à lire
		Problème :
			* "Access is denied"
			* "USN Journal is disabled"
			* La commande ne retourne rien
			* Le fichier généré est vide
			* La commande reste bloquée trop longtemps

		Causes possibles :
			* Terminal non lancé en administrateur
			* Volume différent de C:
			* Disque non formaté en NTFS
			* USN Journal désactivé ou supprimé récemment
			* Journal trop volumineux / PC lent

		Solutions :
			1. Relancer en administrateur
				* Windows + X -> Terminal / Invite de commande (Administrateur)

			2. Vérifier le journal :
				fsutil usn queryjournal C:

			3. Vérifier le type de volume :
				fsutil fsinfo volumeinfo C:

			4. Si le journal est absent ou désactivé :
				⚠️ Ne PAS recréer le journal pendant la vérification
				⚠️ Noter l'information comme red flag possible
				→ Passer à Prefetch, SRUM, Windows.db et Historique des Applications

			5. Si la commande est trop lente :
				* Faire l'export une seule fois dans CMD :
					fsutil usn readjournal C: csv > usn_all.csv
				* Filtrer ensuite localement avec findstr ou Timeline Explorer

	5.a Prefetch vide / désactivé / nettoyé
		Problème :
			* C:\Windows\Prefetch est vide ou presque vide
			* WinPrefetchView n'affiche rien
			* Aucun fichier .pf récent malgré une activité normale du PC
			* La Phase 5 ne permet pas de confirmer l'exécution d'un programme suspect

		⚠️ Règle forensic :
			* NE PAS réactiver Prefetch pendant une vérification en cours
			* NE PAS créer volontairement de nouveaux Prefetch pendant l'analyse
			* Noter l'état actuel comme indice possible de nettoyage / désactivation
			* La réactivation sert uniquement pour les prochaines vérifications

		Étape 1 | Vérifier l'état du dossier Prefetch
			* Windows + X -> Terminal / PowerShell (Administrateur)
			* Taper les commandes :
				Get-Item 'C:\Windows\Prefetch' | Select-Object FullName, LastWriteTime
				Get-ChildItem 'C:\Windows\Prefetch' -Filter *.pf | Measure-Object

			* Interprétation :
				* Dossier vide + LastWriteTime récent
					→ possible nettoyage volontaire
				* Très peu de fichiers .pf sur un PC utilisé
					→ Prefetch possiblement désactivé ou nettoyé
				* Aucun .pf récent alors que Minecraft / logiciels ont été lancés
					→ anomalie à corréler avec SRUM / Windows.db / BAM / Amcache

		Étape 2 | Vérifier la clé registre Prefetch
			* Taper la commande :
				Get-ItemProperty -Path 'HKLM:\SYSTEM\CurrentControlSet\Control\Session Manager\Memory Management\PrefetchParameters' | Select-Object EnablePrefetcher, EnableSuperfetch

			* Valeurs possibles de EnablePrefetcher :
				* 0 → Prefetch désactivé
				* 1 → Prefetch applications uniquement
				* 2 → Prefetch démarrage uniquement
				* 3 → Prefetch applications + démarrage (valeur recommandée)

			⚠️ Si EnablePrefetcher = 0 :
				→ Prefetch désactivé
				→ Noter comme red flag possible si aucune raison légitime

		Étape 3 | Vérifier le service SysMain
			* Taper la commande :
				Get-Service -Name SysMain | Select-Object Name, Status, StartType

			* Interprétation :
				* Status = Running
					→ service actif
				* Status = Stopped ou StartType = Disabled
					→ Prefetch / optimisation Windows potentiellement désactivée

		Étape 4 | Réactivation pour les prochaines vérifications
			⚠️ À faire APRÈS la vérification, pas pendant l'analyse du joueur.
			⚠️ Cette action ne recrée PAS les anciens Prefetch supprimés.
			⚠️ Les nouveaux Prefetch apparaîtront seulement après de futures exécutions de programmes.

			* Réactiver Prefetch côté registre :
				REG ADD "HKLM\SYSTEM\CurrentControlSet\Control\Session Manager\Memory Management\PrefetchParameters" /v EnablePrefetcher /t REG_DWORD /d 3 /f

			* Réactiver SysMain :
				Set-Service -Name SysMain -StartupType Automatic
				Start-Service -Name SysMain

			* Vérifier après réactivation :
				Get-ItemProperty -Path 'HKLM:\SYSTEM\CurrentControlSet\Control\Session Manager\Memory Management\PrefetchParameters' | Select-Object EnablePrefetcher, EnableSuperfetch
				Get-Service -Name SysMain | Select-Object Name, Status, StartType

		Étape 5 | Si Prefetch reste vide après réactivation
			* Lancer un programme bénin après la fin de la vérification
			* Attendre quelques secondes
			* Revérifier :
				Get-ChildItem -Path 'C:\Windows\Prefetch' -Filter *.pf | Select-Object Name, LastWriteTime | Sort-Object LastWriteTime -Descending

			Si aucun .pf n'apparaît :
				→ vérifier les permissions du dossier Prefetch
				→ vérifier les politiques système / outils d'optimisation
				→ vérifier si un cleaner supprime les Prefetch au démarrage

		Conclusion :
			* Prefetch vide pendant la vérification = indice important, mais pas preuve de cheat seul
			* Corréler immédiatement avec :
				- Phase 4 / 4.6 - USN Journal
				- Phase 6 - Amcache
				- Phase 7 - BAM
				- Phase 9 - Windows.db
				- Phase 10 - SRUM

	6.a AmcacheParser / Registry Explorer - utilisation et dépannage
		Problème :
			* Amcache.hve impossible à copier
			* AmcacheParser ne se lance pas
			* Aucun CSV généré
			* CSV vide ou peu exploitable
			* Registry Explorer ne charge pas la ruche
			* Le fichier suspect n'apparaît pas dans Amcache

		Outils utiles :
			* AmcacheParser
				→ parser principal Amcache.hve en CSV
			* Timeline Explorer
				→ lecture / tri / filtrage des CSV
			* Registry Explorer
				→ lecture graphique directe de la ruche
			* 7-Zip
				→ extraction propre des ZIP d'outils

		Installation recommandée :
			1. Télécharger AmcacheParser
				https://download.ericzimmermanstools.com/net9/AmcacheParser.zip

			2. Télécharger Timeline Explorer
				https://download.ericzimmermanstools.com/net9/TimelineExplorer.zip

			3. Télécharger Registry Explorer si besoin
				https://download.ericzimmermanstools.com/net9/RegistryExplorer.zip

			4. Extraire les ZIP dans un dossier simple :
				C:\Tools\EZTools\

			5. Lancer les outils en Administrateur

			⚠️ Si Windows bloque les fichiers téléchargés :
				* Clic droit sur le ZIP -> Propriétés
				* Cocher "Débloquer" si présent
				* Appliquer
				* Extraire ensuite le ZIP

		Méthode rapide avec AmcacheParser :
			* Windows + X -> Terminal / PowerShell (Administrateur)
			* Se placer dans le dossier AmcacheParser :
				cd C:\Tools\EZTools\AmcacheParser

			* Parser la ruche copiée sur le Bureau :
				.\AmcacheParser.exe -f "$env:USERPROFILE\Desktop\VerifAmcache\Amcache.hve" --csv "$env:USERPROFILE\Desktop\VerifAmcache"

			* Parser directement la ruche système si la copie échoue :
				.\AmcacheParser.exe -f "C:\Windows\AppCompat\Programs\Amcache.hve" --csv "$env:USERPROFILE\Desktop\VerifAmcache"

			⚠️ AmcacheParser sait gérer des fichiers verrouillés, mais une copie de travail reste préférable pour conserver la preuve source.

		Erreur : AmcacheParser ne se lance pas
			Causes possibles :
				* Mauvaise version .NET
				* ZIP mal extrait
				* fichier bloqué par Windows
				* antivirus / SmartScreen

			Solutions :
				1. Débloquer le ZIP avant extraction
				2. Extraire avec 7-Zip
				3. Lancer PowerShell en Administrateur
				4. Télécharger le pack à jour via le site Eric Zimmerman
				5. Installer le runtime .NET Desktop si demandé

		Erreur : accès refusé à Amcache.hve
			Solutions :
				1. Relancer PowerShell en Administrateur
				2. Créer un dossier de sortie sur le Bureau
				3. Parser directement :
					.\AmcacheParser.exe -f "C:\Windows\AppCompat\Programs\Amcache.hve" --csv "$env:USERPROFILE\Desktop\VerifAmcache"
				4. Si toujours impossible :
					→ passer à Prefetch / BAM / SRUM / Windows.db
					→ noter l'échec dans le rapport

		Erreur : CSV vide / rien de suspect
			Interprétation possible :
				* Le fichier n'a jamais été inventorié par Amcache
				* Le fichier était trop récent
				* Le fichier était uniquement en mémoire / supprimé très vite
				* Amcache a été nettoyé ou la base est peu alimentée
				* L'exécutable est présent dans un autre artefact mais pas dans Amcache

			Action :
				→ ne pas conclure à l'absence de cheat
				→ vérifier Prefetch, USN, BAM, SRUM, Windows.db

		Utilisation de Timeline Explorer :
			1. Ouvrir Timeline Explorer
			2. Charger le CSV AmcacheParser
			3. Trier par chemin / timestamp
			4. Filtrer les colonnes contenant :
				- Path
				- FullPath
				- LowerCaseLongPath
				- Name
				- SHA1
				- Publisher
			5. Rechercher :
				- C:\Users\
				- AppData
				- Temp
				- Downloads
				- Desktop
				- OneDrive
				- noms suspects

		Utilisation de Registry Explorer :
			1. Ouvrir Registry Explorer en Administrateur
			2. File -> Load Hive
			3. Charger :
				C:\Users\<Utilisateur>\Desktop\VerifAmcache\Amcache.hve
			4. Utiliser Search / Find
			5. Rechercher :
				- nom de fichier
				- chemin
				- SHA1
				- mot-clé suspect

			⚠️ Registry Explorer est surtout utile pour vérifier une entrée précise.
			⚠️ Pour une recherche large, préférer AmcacheParser + Timeline Explorer.

		Champs importants à comprendre :
			* Path / FullPath / LowerCaseLongPath
				→ chemin du fichier connu par Amcache
			* Name / FileName
				→ nom de l'exécutable ou DLL
			* SHA1
				→ hash utilisable pour comparaison
			* Publisher
				→ éditeur déclaré, vide ou inconnu = suspect selon contexte
			* Size
				→ taille du fichier
			* LinkDate
				→ date de compilation PE, peut être falsifiée
			* LastModifiedTime
				→ date de modification connue, à corréler avec les autres artefacts

		⚠️ Pièges fréquents :
			* Confondre présence Amcache et preuve d'exécution
			* Oublier le fuseau horaire
			* Ignorer les chemins utilisateur
			* Se limiter aux noms évidents comme "cheat" ou "vape"
			* Ne pas corréler avec Prefetch / SRUM
			* Conclure "rien" parce qu'Amcache ne contient pas le fichier

	7.a BAM / DAM - utilisation et dépannage
		Problème :
			* Accès refusé à la clé BAM
			* La clé BAM est vide ou absente
			* Plusieurs ControlSet existent
			* Impossible de savoir quel SID correspond à quel utilisateur
			* Timestamp illisible
			* Les chemins sont en \Device\HarddiskVolumeX
			* RECmd ne retourne rien

		Outils utiles :
			* PowerShell
				→ extraction et conversion rapide des timestamps BAM
			* Registry Explorer
				→ lecture graphique de la ruche SYSTEM
			* RECmd
				→ export registre avancé en CSV
			* Timeline Explorer
				→ lecture du CSV bam_parsed.csv

		Téléchargements utiles :
			* Registry Explorer :
				https://download.ericzimmermanstools.com/net9/RegistryExplorer.zip
			* RECmd :
				https://download.ericzimmermanstools.com/net9/RECmd.zip
			* Timeline Explorer :
				https://download.ericzimmermanstools.com/net9/TimelineExplorer.zip

		Étape 1 | Vérifier l'existence de BAM
			* Windows + X -> Terminal / PowerShell (Administrateur)
			* Taper :
				Test-Path 'HKLM:\SYSTEM\CurrentControlSet\Services\bam\State\UserSettings'

			* Vérifier l'ancien chemin BAM :
				Test-Path 'HKLM:\SYSTEM\CurrentControlSet\Services\bam\UserSettings'

			* Vérifier DAM :
				Test-Path 'HKLM:\SYSTEM\CurrentControlSet\Services\dam\State\UserSettings'

			Interprétation :
				* True
					→ clé présente
				* False
					→ clé absente, vérifier version Windows / droits / nettoyage possible
				* `bam\State\UserSettings` ou `bam\UserSettings`
					→ selon la version Windows, l'un ou l'autre peut être utilisé

		Étape 2 | Si accès refusé
			Solutions :
				1. Relancer PowerShell en Administrateur
				2. Ouvrir Registry Explorer en Administrateur
				3. Copier la ruche SYSTEM puis l'analyser hors-ligne :
					Copy-Item "C:\Windows\System32\config\SYSTEM" -Destination "$env:USERPROFILE\Desktop\VerifBAM\SYSTEM" -Force
				4. Charger la copie dans Registry Explorer

			⚠️ Ne pas modifier les permissions registre pendant la vérification.

		Étape 3 | Identifier le bon ControlSet
			* CurrentControlSet correspond au ControlSet actif
			* Dans une ruche SYSTEM chargée hors-ligne, utiliser :
				HKLM\SYSTEM\Select

			* Lire la valeur Current :
				reg query HKLM\SYSTEM\Select

			* Interprétation :
				* Current = 1
					→ utiliser ControlSet001
				* Current = 2
					→ utiliser ControlSet002

			* Dans Registry Explorer :
				ControlSet00X\Services\bam\State\UserSettings
				ControlSet00X\Services\bam\UserSettings

		Étape 4 | Traduire un SID en utilisateur
			* Exemple PowerShell :
				$sid='S-1-5-21-0000000000-0000000000-0000000000-1001'
				([Security.Principal.SecurityIdentifier]$sid).Translate([Security.Principal.NTAccount]).Value

			* Si la traduction échoue :
				→ compte supprimé
				→ profil non chargé
				→ SID système / service
				→ noter le SID brut

			* SIDs fréquents :
				- S-1-5-18		→ SYSTEM
				- S-1-5-19		→ LOCAL SERVICE
				- S-1-5-20		→ NETWORK SERVICE
				- S-1-5-21-...	→ utilisateur local / domaine

		Étape 5 | Comprendre les chemins \Device\HarddiskVolumeX
			* BAM affiche souvent :
				\Device\HarddiskVolume4\Users\Nom\Downloads\loader.exe

			* Cela correspond généralement à un volume Windows type C:
			* Ne pas bloquer l'analyse sur la conversion exacte
			* Ce qui compte :
				- présence de \Users\
				- dossier Downloads / Desktop / Temp / AppData
				- nom du fichier
				- timestamp
				- utilisateur SID

		Étape 6 | Si RECmd ne retourne rien
			Causes possibles :
				* mauvais ControlSet
				* ruche SYSTEM non copiée correctement
				* clé BAM absente
				* option --kn pointant sur une mauvaise clé

			Solutions :
				1. Tester ControlSet001 puis ControlSet002
				2. Ouvrir la ruche avec Registry Explorer pour vérifier le chemin
				3. Revenir à la méthode PowerShell live
				4. Corréler avec Prefetch / SRUM / Amcache si BAM est vide

		Étape 7 | Si BAM est vide
			Interprétation possible :
				* nettoyage système
				* artefact non alimenté
				* exécution trop ancienne
				* programme non enregistré par BAM
				* analyse faite sur mauvais ControlSet

			Action :
				→ ne pas conclure à l'absence de cheat
				→ passer à Prefetch, Amcache, SRUM, Windows.db, USN

		Champs importants :
			* SID
				→ utilisateur
			* Path
				→ programme concerné
			* LastRunLocal
				→ timestamp local décodé depuis FILETIME

		⚠️ Pièges fréquents :
			* Confondre BAM avec Prefetch
			* Conclure sur BAM seul
			* Ignorer les chemins \Device\HarddiskVolumeX
			* Oublier de traduire le SID
			* Utiliser le mauvais ControlSet
			* Ne pas comparer avec l'heure de session Minecraft

	8.a NTUSER.DAT / UserAssist - utilisation et dépannage
		Problème :
			* NTUSER.DAT impossible à copier
			* Registry Explorer ne charge pas la ruche
			* RECmd ne retourne rien
			* UserAssist est illisible ou encodé
			* RecentDocs / RunMRU sont vides
			* Impossible de savoir quel NTUSER appartient au joueur

		Outils utiles :
			* Registry Explorer
				→ lecture graphique de NTUSER.DAT / UsrClass.dat
			* RECmd
				→ export de clés registre en CSV
			* UserAssistView
				→ lecture rapide de UserAssist
			* RecentFilesView / LastActivityView
				→ complément activité utilisateur

		Téléchargements utiles :
			* Registry Explorer :
				https://download.ericzimmermanstools.com/net9/RegistryExplorer.zip
			* RECmd :
				https://download.ericzimmermanstools.com/net9/RECmd.zip
			* UserAssistView :
				https://www.nirsoft.net/utils/userassistview.zip
			* RecentFilesView :
				https://www.nirsoft.net/utils/recentfilesview.zip

		Étape 1 | Si NTUSER.DAT est verrouillé
			* Méthode recommandée pour l'utilisateur courant :
				reg save HKCU "$env:USERPROFILE\Desktop\VerifNTUSER\NTUSER_HKCU.hiv" /y

			* Pour un autre utilisateur connecté :
				1. Relever son SID dans :
					HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\ProfileList
				2. Sauvegarder la ruche chargée :
					reg save HKU\<SID> "$env:USERPROFILE\Desktop\VerifNTUSER\NTUSER_<SID>.hiv" /y

			⚠️ Remplacer <SID> par le SID réel.
			⚠️ Ne pas prendre possession de NTUSER.DAT pendant une vérification sauf nécessité absolue.

		Étape 2 | Si UsrClass.dat est verrouillé
			* Essayer la copie simple :
				Copy-Item "$env:LOCALAPPDATA\Microsoft\Windows\UsrClass.dat" -Destination "$env:USERPROFILE\Desktop\VerifNTUSER\UsrClass.dat" -Force

			* Si impossible :
				→ continuer avec NTUSER.DAT
				→ noter que UsrClass.dat n'a pas pu être extrait
				→ corréler avec Windows.db / RecentFilesView / ShellBags si besoin

		Étape 3 | Si UserAssist est illisible
			* UserAssist encode souvent les noms en ROT13
			* Utiliser UserAssistView pour une lecture directe
			* Avec Registry Explorer :
				Software\Microsoft\Windows\CurrentVersion\Explorer\UserAssist\...\Count

			Interprétation :
				* Run Count
					→ nombre de lancements GUI
				* Last Run Time
					→ dernière exécution via interface graphique
				* Program Name / Path
					→ programme lancé

		Étape 4 | Si RECmd ne retourne rien
			Causes possibles :
				* mauvaise ruche chargée
				* mauvaise clé
				* profil utilisateur incorrect
				* clé vide / nettoyée

			Solutions :
				1. Vérifier la ruche avec Registry Explorer
				2. Vérifier le chemin exact de la clé
				3. Tester une clé simple comme RunMRU
				4. Utiliser UserAssistView / RecentFilesView en complément

		Étape 5 | Si RecentDocs / RunMRU sont vides
			Interprétation possible :
				* historique nettoyé
				* utilisateur peu actif
				* activité faite via ligne de commande / loader direct
				* politique Windows limitant les historiques

			Action :
				→ ne pas conclure à l'absence de cheat
				→ vérifier UserAssist, BAM, Prefetch, Windows.db, SRUM

		Champs / clés importants :
			* RecentDocs
				→ fichiers récemment manipulés
			* RunMRU
				→ commandes Windows + R
			* UserAssist
				→ lancements via GUI
			* ComDlg32
				→ fichiers sélectionnés dans boîtes Ouvrir / Enregistrer
			* TypedPaths
				→ chemins tapés dans l'explorateur

		⚠️ Pièges fréquents :
			* Conclure sur RecentDocs seul
			* Oublier que UserAssist est encodé
			* Analyser le mauvais profil utilisateur
			* Ignorer UsrClass.dat
			* Oublier de comparer avec l'heure Minecraft
			* Confondre fichier sélectionné et fichier exécuté

	9.a Windows.db - extraction, parsing et dépannage
		Problème :

			* Accès refusé au dossier Windows Search
			* Parser Windows.db ne fonctionne pas
			* CSV énorme / Timeline Explorer lent
			* Données fragmentées difficiles à comprendre
			* Aucun résultat suspect visible

		Outils utiles :
			* Windows Database Parser
				→ parser Windows.db en données exploitables
			* Timeline Explorer
				→ lecture / tri / regroupement des CSV
			* esentutl
				→ copie d'un fichier verrouillé
			* PowerShell
				→ création de dossier / copie / filtrage

				→ lecture / filtrage des CSV

			* EvtxECmd :
				https://download.ericzimmermanstools.com/net9/EvtxECmd.zip
			* Timeline Explorer :
				https://download.ericzimmermanstools.com/net9/TimelineExplorer.zip

			* Définir le dossier de sortie :
				$Desktop = [Environment]::GetFolderPath("Desktop")
				$OutDir = Join-Path $Desktop "VerifEventLogs"
				New-Item -ItemType Directory -Path $OutDir -Force

			* Exporter avec wevtutil plutôt que copier manuellement :
				wevtutil epl Security (Join-Path $OutDir "Security.evtx") /ow:true

			⚠️ Éviter de prendre possession des fichiers EVTX pendant une vérification sauf nécessité absolue.

		Étape 2 | Si un canal EventLog n'existe pas
			* Lister les canaux disponibles :
				wevtutil el | findstr /i "PowerShell TaskScheduler DNS Sysmon"

			* Interprétation :
				* Sysmon absent
					→ normal si jamais installé
				* PowerShell Operational absent / désactivé
					→ moins de détails sur scripts
				* DNS Operational absent
					→ passer à cache DNS / navigateur / réseau

		Étape 3 | Si 4688 est absent
			* 4688 = création de processus dans Security.evtx
			* Il n'apparaît que si l'audit "Process Creation" est activé

			* Vérifier l'état :
				auditpol /get /subcategory:"Process Creation"

			* Pour les prochaines vérifications uniquement :
				auditpol /set /subcategory:"Process Creation" /success:enable /failure:enable

			⚠️ Ne pas activer pendant une vérification en cours pour interpréter le passé.
			⚠️ L'activation ne recrée pas les anciens événements.

		Étape 4 | Si PowerShell Operational est vide
			Interprétation possible :
				* script block logging non activé
				* activité trop ancienne
				* logs nettoyés
				* PowerShell non utilisé

			* Pour les prochaines vérifications uniquement :
				REG ADD "HKLM\SOFTWARE\Policies\Microsoft\Windows\PowerShell\ScriptBlockLogging" /v EnableScriptBlockLogging /t REG_DWORD /d 1 /f

			⚠️ Cette activation sert au futur, pas au passé.

		Étape 5 | Si EvtxECmd ne génère rien
			Solutions :
				1. Parser des EVTX exportés, pas les logs live
				2. Vérifier que les fichiers .evtx existent
				3. Utiliser un chemin simple :
					C:\Tools\EvtxECmd\
				4. Créer le dossier CSV :
					$Desktop = [Environment]::GetFolderPath("Desktop")
					$OutDir = Join-Path $Desktop "VerifEventLogs"
					New-Item -ItemType Directory -Path (Join-Path $OutDir "CSV") -Force
				5. Relancer :
					.\EvtxECmd.exe -d $OutDir --csv (Join-Path $OutDir "CSV")

		Étape 6 | Si les logs sont énormes
			* Dans Timeline Explorer :
				- désactiver Auto Size Columns
				- filtrer par TimeCreated
				- filtrer par Id
				- filtrer par Message
				- ouvrir un CSV à la fois

			* Prioriser :
				- plage horaire Minecraft
				- 30 minutes avant / après
				- mots-clés suspects

		Étape 7 | Si un journal a été vidé
			* Indices :
				- Security ID 1102
				- Security ID 1100
				- Security ID 1104
				- System ID 104 selon contexte
				- Application ID 3079 selon Provider / Message
				- EventLog service redémarré
				- gros trou temporel dans les logs

			Action :
				→ noter comme red flag anti-forensic
				→ corréler avec USN, Prefetch, SRUM, Windows.db, MFT
				→ ne pas conclure seul sans autres traces

		Étape 8 | Événements à forte valeur de corrélation
			* Heure système modifiée :
				- Security 4616
				→ vérifier si les timestamps des artefacts deviennent incohérents
				→ comparer avec Prefetch / USN / MFT / SRUM

			* Périphériques externes :
				- Security 6416 → nouveau périphérique reconnu
				- Security 6420 → périphérique désactivé
				→ utile si cheat / loader lancé depuis clé USB ou disque externe

			* Services :
				- Security 4697 → service installé
				- System 7045 → service installé
				→ relever nom du service, chemin, compte, heure

			* Processus :
				- Security 4688 → processus créé
				- Security 4689 → processus terminé
				→ seulement si audit Process Creation activé
				→ vérifier si la ligne de commande est présente

			* Groupes / permissions :
				- Security 4798
				- Security 4799
				→ indique une énumération d'appartenance à des groupes locaux
				→ ne pas interpréter comme suppression de clichés ou modification Regedit seule

		⚠️ Pièges fréquents :
			* Attendre que les EventLogs prouvent tout
			* Oublier que Sysmon n'est pas installé par défaut
			* Confondre absence de 4688 et absence d'exécution
			* Ignorer PowerShell 4103 / 4104
			* Oublier les logs vidés comme red flag
			* Confondre 4798 / 4799 avec une action de suppression
			* Interpréter 3079 sans lire le Provider / Message complet

DNS vidé ou inaccessible
		Problème : La commande Get-DnsClientCache ne retourne rien ou erreur "Aucune entrée trouvée"
		

	13.a Cache 
		Cause :
			Le joueur a vidé le cache DNS manuellement
			Service DNS Client désactivé
			Cache DNS automatiquement vidé par un cleaner
		
		Solutions :
			1. Vérifier si le service DNS Client est actif
					powershellGet-Service -Name "Dnscache" | Select-Object Name, Status, StartType
				Si le service est arrêté :
					powershellStart-Service -Name "Dnscache"
					Set-Service -Name "Dnscache" -StartupType Automatic
			
			2. Vérifier les EventLogs DNS
				Même si le cache est vidé, les requêtes peuvent être dans les logs :
					powershellGet-WinEvent -LogName "Microsoft-Windows-DNS-Client/Operational" | Where-Object {$_.TimeCreated -gt (Get-Date).AddHours(-24)} | Select-Object TimeCreated, Message | Out-GridView
			
			3. Analyser le fichier hosts (possible redirection)
					powershellGet-Content C:\Windows\System32\drivers\etc\hosts
				⚠️ Si le cache DNS est vide ET le service actif → Suspicion de nettoyage volontaire (red flag)

	13.b Wireshark bloqué ou non installable
		Problème : Installation de Wireshark refusée ou bloquée par l'antivirus/contrôle parental
			Solutions alternatives :
				1. Utiliser netsh trace (natif Windows)
					powershell# Démarrer la capture réseau
					netsh trace start capture=yes tracefile=C:\capture.etl
				# Laisser tourner pendant l'activité suspecte (2-3 min)
				
					netsh trace stop
				Analyser le fichier .etl :
					Installer Microsoft Message Analyzer ou Network Monitor
					Ou convertir en .pcap avec : https://github.com/microsoft/etl2pcapng

				2. Utiliser TCPView (plus léger, portable)
					Télécharger la version portable (pas d'installation)
					Lancer directement tcpview.exe

				3. PowerShell - Monitoring réseau basique
					powershell# Surveiller les connexions en temps réel
					while ($true) {
						Get-NetTCPConnection | Where-Object {$_.State -eq "Established"} | Select-Object LocalAddress, RemoteAddress, RemotePort, OwningProcess, @{Name="Process";Expression={(Get-Process -Id $_.OwningProcess -ErrorAction SilentlyContinue).ProcessName}} | Format-Table
						Start-Sleep -Seconds 2
						Clear-Host
					}
					# Arrêter la surveillance (Ctrl+C pour arrêter)

	13.c TCPView vide ou n'affiche rien
		Problème : TCPView ne montre aucune connexion ou erreur "Access Denied"
		Causes :
			TCPView non lancé en administrateur
			Pare-feu Windows bloque l'outil
			Processus masqués par un rootkit

		Solutions :
			1. Lancer en administrateur
				Clic droit → Exécuter en tant qu'administrateur

			2. Désactiver temporairement le pare-feu (pour test)
					powershell# Désactiver le pare-feu (temporaire)
					Set-NetFirewallProfile -Profile Domain,Public,Private -Enabled False
				* Relancer TCPView
				* Réactiver le pare-feu IMMÉDIATEMENT après
					Set-NetFirewallProfile -Profile Domain,Public,Private -Enabled True
				
			3. Vérifier avec netstat en fallback
				powershellnetstat -ano 5
				(Rafraîchit toutes les 5 secondes)

	13.d PEStudio / Detect It Easy - Fichier non analysable
		Problème : L'outil refuse d'analyser le fichier ou crash
		Causes :
			Fichier corrompu
			Fichier protégé par anti-debug
			Format non supporté

		Solutions :
			1. Vérifier l'intégrité du fichier
				* Calculer le hash SHA256
					Get-FileHash -Path "C:\chemin\vers\fichier.exe" -Algorithm SHA256
				* Vérifier sur VirusTotal : https://www.virustotal.com
			
			2. Contourner l'anti-debug (PEStudio)
				* Ouvrir PEStudio
					* Aller dans Settings
					* Décocher Enable Anti-Virus Check
					* Réessayer

			3. Alternative : Utiliser strings.exe uniquement
				strings.exe -a -n 8 "C:\chemin\vers\fichier.exe" | Select-String -Pattern "inject|hook|bypass|cheat|kernel32|VirtualAlloc|WriteProcessMemory"

	13.e BinDiff - Impossible de comparer (versions différentes)
		Problème : BinDiff indique "Version mismatch" ou "No matching functions"
		Causes :
			Les deux fichiers sont de versions différentes
			Architecture différente (x86 vs x64)
			Compilation différente (optimisations)

		Solutions :
			1. Vérifier la version exacte
				powershell# Afficher les métadonnées du fichier
				Get-Item "C:\chemin\vers\fichier.exe" | Select-Object Name, VersionInfo
				Télécharger la version EXACTE depuis la source officielle
				
			2. Comparer manuellement les imports critiques
				* Dans IDA / Ghidra :
					* Ouvrir Imports (View → Imports)
					* Rechercher les fonctions suspectes :
						kernel32.dll : VirtualAllocEx, WriteProcessMemory, CreateRemoteThread
						ntdll.dll : NtWriteVirtualMemory, NtCreateThreadEx
						user32.dll : SetWindowsHookEx, GetAsyncKeyState

			3. Alternative : Comparaison par sections
					powershell# Comparer la taille des sections PE
					$suspect = [System.Reflection.Assembly]::LoadFile("C:\chemin\suspect.exe")
					$legitime = [System.Reflection.Assembly]::LoadFile("C:\chemin\legitime.exe")

				* Si les tailles diffèrent = modification

	13.f Strings.exe - Commande non reconnue
		Problème : strings.exe : Le terme 'strings.exe' n'est pas reconnu
		Cause : Sysinternals Strings non installé ou non dans le PATH
		Solutions :
			1. Télécharger Strings.exe
				Lien direct : https://live.sysinternals.com/strings.exe
				Ou pack complet : https://download.sysinternals.com/files/SysinternalsSuite.zip

			2. Utiliser le chemin complet
				powershell# Exemple si téléchargé sur le Bureau
				C:\Users\<Utilisateur>\Desktop\strings.exe -a -n 8 "C:\chemin\fichier.exe" > strings.txt
				
			3. Alternative PowerShell (moins efficace)
				powershell# Extraction basique de strings (ASCII uniquement)
				Select-String -Path "C:\chemin\fichier.exe" -Pattern '[a-zA-Z0-9]{8,}' | Select-Object -ExpandProperty Line

	13.g Pare-feu bloque toutes les connexions sortantes
		Problème : TCPView/Wireshark ne détecte aucune connexion car le pare-feu est trop restrictif
		Solutions :
			1. Vérifier les règles du pare-feu
				powershellGet-NetFirewallRule | Where-Object {$_.Direction -eq "Outbound" -and $_.Action -eq "Block"} | Select-Object DisplayName, Enabled
			
			2. Créer une règle temporaire pour javaw.exe
				powershell# Autoriser javaw.exe temporairement
				New-NetFirewallRule -DisplayName "Allow javaw.exe" -Direction Outbound -Program "C:\Program Files\Java\...\javaw.exe" -Action Allow

			# Supprimer après analyse
				Remove-NetFirewallRule -DisplayName "Allow javaw.exe"
			
			3. Analyser les logs du pare-feu
				powershell# EventLog du pare-feu
				Get-WinEvent -LogName "Microsoft-Windows-Windows Firewall With Advanced Security/Firewall" | Where-Object {$_.Id -eq 5157} | Select-Object TimeCreated, Message | Out-GridView

	13.h Accès refusé aux fichiers système (EventLogs, SRUM, etc.)
		Problème : Get-WinEvent : Accès refusé ou Copy-Item : Permission denied
		Cause : PowerShell non lancé en administrateur ou fichiers verrouillés
		Solutions :
			1. Relancer PowerShell en Administrateur
				Windows + X → Terminal (Admin)

			2. Prendre possession du fichier (si verrouillé)
				powershell# Exemple pour un fichier EventLog
				takeown /f C:\Windows\System32\winevt\Logs\Application.evtx
				icacls C:\Windows\System32\winevt\Logs\Application.evtx /grant Administrateurs:F
			
			3. Copier les fichiers au lieu de les lire directement
				powershell# Copier SRUDB.dat vers le Bureau
				Copy-Item "C:\Windows\System32\sru\SRUDB.dat" -Destination "$env:USERPROFILE\Desktop\SRUDB.dat" -Force

	13.i Service "DNS Client" désactivé volontairement
		Problème : Get-Service Dnscache retourne "Stopped" et impossible de redémarrer
		Cause : Service désactivé manuellement (red flag majeur)
		Vérification :
				powershellGet-Service -Name "Dnscache" | Select-Object Name, Status, StartType
			Si StartType = "Disabled" :
				powershell# Réactiver le service
				Set-Service -Name "Dnscache" -StartupType Automatic
				Start-Service -Name "Dnscache"
		⚠️ Si le joueur refuse de réactiver → Suspicion élevée (possible nettoyage anti-forensic)

	13.j Wireshark capture vide (0 paquets)
		Problème : La capture Wireshark ne montre aucun paquet
		Causes :
			Mauvaise interface réseau sélectionnée
			Pare-feu bloque la capture
			Driver WinPcap/Npcap manquant
		Solutions :
			1. Vérifier l'interface réseau active
			powershellGet-NetAdapter | Where-Object {$_.Status -eq "Up"} | Select-Object Name, InterfaceDescription
		
			2. Réinstaller Npcap
			Télécharger : https://npcap.com/#download
			Installer avec option WinPcap Compatibility Mode
			
			3. Lancer Wireshark en mode promiscuous
				Capture → Options → Cocher Enable promiscuous mode

	20.a Bypass Hardlink / Taskmgr / Conhost - dépannage et faux positifs
		Problème :
			* Le hardlink temporaire a déjà été supprimé
			* USN ne montre pas clairement "Hard Link Change"
			* fsutil ne montre pas le chemin complet
			* JournalTrace montre un Current file différent du Name
			* javaw.security est supprimé mais BLUE.pf reste visible
			* audiodg.exe apparaît mais peut être légitime
			* La DLL dans System32 semble cachée ou récente
			* La tâche planifiée a été supprimée avant vérification
			* conhost / taskmgr ne donnent pas de preuve directe
			* Plusieurs DLL suspectes existent dans System32

		Règle importante :
			* NE PAS exécuter la DLL suspecte
			* NE PAS recréer le hardlink sur la machine du joueur
			* NE PAS baser la conclusion sur le nom audiodg.exe seul
			* Toujours corréler :
				→ JournalTrace / USN
				→ Name + Current file
				→ heure Minecraft
				→ SRUM / BAM / Prefetch si présents
				→ mémoire javaw.exe si possible

		Étape 1 | Si le hardlink existe encore
			* Vérifier les liens durs actuels :
				fsutil hardlink list "%TEMP%\NomSuspect.exe"

			* Vérifier aussi depuis la DLL suspecte :
				fsutil hardlink list "C:\Windows\System32\NomSuspect.dll"

			Interprétation :
				* plusieurs chemins listés
					→ hardlink encore présent
				* un seul chemin listé
					→ le lien temporaire peut déjà avoir été supprimé
				* erreur fichier introuvable
					→ passer à USN / MFT / Prefetch

		Étape 2 | Si le hardlink est supprimé
			* Chercher les traces indirectes :
				- USN : Hard Link Change / Close
				- MFT : RecordModified proche de l'heure suspecte
				- Prefetch : NOMEXE.EXE-*.pf
				- BAM : chemin \AppData\Local\Temp\
				- SRUM : activité du nom .exe proche de javaw.exe
				- Amcache : nom .exe / DLL si inventorié

			⚠️ La suppression du lien temporaire n'efface pas forcément USN / Prefetch / BAM / SRUM.

		Étape 3 | Si USN ne montre pas le texte "Hard Link Change"
			* Selon l'outil, la raison peut apparaître :
				- en texte
				- en code hexadécimal
				- combinée avec Close

			* Rechercher aussi :
				- 0x00010000
				- 0x80010000
				- Close
				- le nom .exe temporaire
				- \Temp\
				- le nom de la DLL

			* Si USN est vide / désactivé :
				→ Voir ANNEXE 4.a
				→ passer à MFT / Prefetch / BAM / SRUM

		Étape 3.b | Si le chemin n'est pas lisible dans fsutil
			* C'est normal :
				→ fsutil usn readjournal affiche le nom du fichier
				→ fsutil affiche aussi FileRef / ParentRef
				→ fsutil ne reconstruit pas toujours le chemin complet

			* Méthode recommandée :
				1. Ouvrir JournalTrace
				2. Filtrer Reason = Hard Link / Hard link change
				3. Double-cliquer l'entrée
				4. Lire :
					- Name
					- Directory
					- Current file
					- File Identification
					- Date

			* Exemple :
				- Name = javaw.security
				- Directory = C:\Program Files\Eclipse Adoptium\...\jre\lib\security
				- Current file = C:\Users\<Utilisateur>\AppData\Roaming\PrismLauncher\java\jre-legacy\bin\BLUE.pf
				→ le hardlink supprimé est javaw.security
				→ le fichier courant / lié est BLUE.pf

			* Si un outil type Journal Parser / OrbDiff affiche Directory :
				→ utiliser Directory pour confirmer le chemin
				→ mais conserver aussi l'export fsutil brut

			* Si JournalTrace n'est pas disponible :
				1. Noter Name et ParentRef dans l'export fsutil
				2. Parser la MFT avec Phase 12 - MFTECmd
				3. Chercher le même Name dans mft_C.csv
				4. Comparer ParentPath / FullPath / EntryNumber selon les colonnes

		Étape 3.c | Cas javaw.security supprimé / BLUE.pf restant
			* Interprétation :
				- javaw.security = alias hardlink supprimé après selfdestruct
				- BLUE.pf = fichier courant / payload déguisé
				- .pf hors C:\Windows\Prefetch = extension trompeuse possible

			* Action :
				1. Relever Current file dans JournalTrace
				2. Vérifier si BLUE.pf existe encore
				3. Calculer hash SHA256
				4. Vérifier signature
				5. Vérifier si le fichier commence par MZ
				6. Corréler Date JournalTrace avec Minecraft / SRUM / BAM / NTUSER

			⚠️ Si BLUE.pf ne laisse aucune trace d'exécution classique :
				→ la preuve principale devient JournalTrace
				→ renforcer avec MFT / USN / Windows.db / mémoire javaw.exe

		Étape 4 | Cas audiodg.exe
			* audiodg.exe légitime existe normalement dans :
				C:\Windows\System32\audiodg.exe

			* Suspect si :
				- audiodg.exe apparaît dans %TEMP%
				- audiodg.exe a un Prefetch différent proche de Minecraft
				- audiodg.exe apparaît dans BAM avec chemin utilisateur / Temp
				- USN montre Hard Link Change sur audiodg.exe dans Temp
				- une DLL type audio*.dll non signée apparaît dans System32

			⚠️ audiodg.exe dans System32 seul = normal.
			⚠️ audiodg.exe dans Temp + hardlink = anomalie forte.

		Étape 5 | Si plusieurs DLL System32 sont suspectes
			* Prioriser :
				1. DLL créée / modifiée le jour de la vérification
				2. DLL non signée ou signature invalide
				3. DLL avec nom proche d'un composant Windows mais légèrement différent
				4. DLL dont la date correspond au Hard Link Change USN
				5. DLL mentionnée dans Prefetch / Amcache / Windows.db

			* Pour chaque DLL prioritaire :
				Get-FileHash "C:\Windows\System32\NomSuspect.dll" -Algorithm SHA256
				Get-AuthenticodeSignature "C:\Windows\System32\NomSuspect.dll" | Format-List

			* Si disponible :
				→ vérifier avec Sigcheck
				→ vérifier strings
				→ vérifier VirusTotal par hash uniquement

		Étape 6 | Si la tâche planifiée est supprimée
			* Vérifier les traces restantes :
				- C:\Windows\System32\Tasks
				- registre TaskCache
				- EventLogs TaskScheduler
				- Security 4698 / 4702 / 4699 si audit activé
				- USN / MFT sur le dossier Tasks

			* Commande TaskCache :
				Get-ChildItem 'HKLM:\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Schedule\TaskCache\Tree' -Recurse -ErrorAction SilentlyContinue | Select-Object Name, PSChildName

			* Commande fichiers Tasks :
				Get-ChildItem "C:\Windows\System32\Tasks" -Recurse -Force | Select-Object FullName, CreationTime, LastWriteTime, Length | Sort-Object LastWriteTime -Descending

			* Pour lire les LastWriteTime registre TaskCache :
				→ utiliser Registry Explorer ou RECmd sur la ruche SOFTWARE

			* EventLogs utiles :
				- TaskScheduler 106 → tâche créée
				- TaskScheduler 140 → tâche modifiée
				- TaskScheduler 200 → action lancée
				- TaskScheduler 201 → action terminée
				- Security 4698 → tâche créée
				- Security 4702 → tâche modifiée
				- Security 4699 → tâche supprimée

		Étape 7 | Si conhost / taskmgr ne prouvent rien
			* C'est normal :
				- conhost.exe est courant
				- taskmgr.exe peut être lancé légitimement
				- les fenêtres fermées ne donnent pas toujours une preuve directe

			* Chercher plutôt :
				- taskmgr lancé automatiquement au logon
				- taskmgr / cmd / conhost proche du Hard Link Change
				- tâche planifiée RunLevel Highest
				- Prefetch TASKMGR.EXE / CMD.EXE / CONHOST.EXE proche de la session
				- SRUM montrant activité console proche de Minecraft

		Étape 8 | Barème de conclusion
			* Faible :
				→ DLL non signée dans System32 seule
			* Moyen :
				→ DLL suspecte + nom .exe en Temp dans USN
			* Fort :
				→ Hard Link Change + DLL System32 suspecte + Prefetch du faux .exe
			* Très fort :
				→ Hard Link Change + DLL suspecte + BAM/SRUM pendant Minecraft + tâche planifiée au logon
			* Direct :
				→ strings cheat dans javaw.exe / preuve mémoire positive

		Étape 9 | Pour les prochaines vérifications
			⚠️ À activer uniquement pour améliorer les futures détections.
			⚠️ Ces actions ne recréent pas les traces du passé.

			* Activer l'audit création de processus :
				auditpol /set /subcategory:"Process Creation" /success:enable /failure:enable

			* Inclure les lignes de commande dans les EventLogs 4688 :
				REG ADD "HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Policies\System\Audit" /v ProcessCreationIncludeCmdLine_Enabled /t REG_DWORD /d 1 /f

			* Vérifier que TaskScheduler Operational est activé :
				wevtutil sl Microsoft-Windows-TaskScheduler/Operational /e:true

			* Option avancée :
				→ installer Sysmon sur les machines que vous contrôlez
				→ surveiller création de processus, fichiers créés et tâches planifiées
				→ ne pas installer pendant une vérification joueur si cela modifie trop le contexte

		⚠️ Pièges fréquents :
			* Confondre audiodg.exe légitime et faux audiodg.exe en Temp
			* Conclure sur une DLL non signée seule
			* Relancer la commande suspecte et modifier les preuves
			* Oublier les tâches planifiées supprimées
			* Ignorer USN / MFT sur C:\Windows\System32\Tasks
			* Ne pas comparer avec l'heure Minecraft

			* Lien : https://live.sysinternals.com/Coreinfo.exe (32-bit) / https://live.sysinternals.com/Coreinfo64.exe (64-bit)
			* Fonctionnalité : Affiche infos CPU, virtualisation et sécurité (SMEP, SMAP, DEP, Hyper-V)
			* Utilité en Forensic :
				→ Détecter des désactivations de protections kernel
				→ Vérifier si le PC est en VM
				→ Identifier des configurations de bypass
			* Type de cheat/bypass recherché :
				→ SMEP/SMAP désactivés (pour injection kernel)
				→ DEP désactivé (pour exploitation mémoire)
				→ Virtualisation détectée (si jeu interdit en VM)
				→ Protections système compromises
			* Commande utile :
				→ cmdcoreinfo.exe

	Phase 21 – Bypass CMD « Package » (type Bruzen) – DÉTECTION

		Objectif :
			* Détecter un bypass exécutant CMD en tant que « package » pour masquer un cheat
			* Repérer le remplacement (replace) d'un cheat par un programme légitime
			* Corréler les traces avec l'heure de jeu / lancement Minecraft

		Principe de la méthode (pour comprendre quoi chercher) :
			* CMD est lancé sous forme de « package » via Microsoft.WindowsTerminal, avec privilèges administrateur
			* Un fichier cheat est temporairement renommé / remplacé par le nom d'un programme légitime
			* Une fois la vérification passée, le fichier original est restauré
			* Résultat : la personne qui inspecte le fichier ne voit qu'un programme légitime
			⚠️ Cette phase ne sert qu'à DÉTECTER la méthode, jamais à la reproduire.

		────────────────────────────────────────────
		Indicateur 1 | Traces dans BAM

			* L'ouverture de CMD en « package » nécessite Microsoft.WindowsTerminal
			* Cette opération laisse une trace WindowsTerminal dans BAM, avec l'heure exacte
			* Consultation : → Voir Phase 7 – BAM (consulter via Regedit ou WinLiveInfo)
			
			⚠️ Interprétation prudente :
				- Une entrée WindowsTerminal seule N'EST PAS une preuve
				- Elle peut provenir d'une utilisation légitime du terminal
				- Plus suspect uniquement si WindowsTerminal apparaît dès le démarrage du PC
				  (Windows ne l'enregistre pas naturellement à cet instant)
			
			(Réf. image : https://imgur.com/a/5HHGHrK)

		────────────────────────────────────────────
		Indicateur 2 | Traces dans le Kernel

			* Un dump du Kernel peut parfois contenir une partie (voire la totalité) de la commande utilisée :
				- la commande d'ouverture de CMD en package
				- la commande exécutée à l'intérieur du CMD
			* Procédure : → Voir Phase 12 – MFT (section Analyse mémoire Kernel)
			
			⚠️ Souvent rien n'est présent : l'absence de trace ne disculpe pas, mais une trace est un indice fort.
			
			(Réf. image : https://imgur.com/a/oDtYsYX)

		────────────────────────────────────────────
		Indicateur 3 | Traces dans DPS

			* Un dump DPS filtré sur « Microsoft.WindowsTerminal » peut révéler des traces de l'opération
			* Aucun résultat = bon élément pour ÉCARTER ce bypass
			* Si résultats présents, analyser attentivement :
				- une entrée Microsoft.WindowsTerminal suivie d'un exécutable suspect
				  → l'exécutable peut être le programme légitime servant à masquer le cheat
				- la présence de conhost.exe
				  → généralement lié au CMD exécuté en tant que package
			
			(Réf. image : https://imgur.com/a/k8nUcZ3)

		────────────────────────────────────────────
		Indicateur 4 | Traces dans DiagTrack / TextInputHost

			* DiagTrack et des processus comme TextInputHost peuvent parfois contenir des fragments des commandes
			* Mots-clés à rechercher :
				- Microsoft.WindowsTerminal
				- WindowsTerminal
				- conhost
			
			⚠️ Aide LIMITÉE : ces informations disparaissent très vite et n'apportent souvent rien.
			   Mais quand ça fonctionne, c'est une preuve complémentaire intéressante.
			
			(Réf. image : https://imgur.com/a/FVsHrRj)

		────────────────────────────────────────────
		Indicateur 5 | Data Truncation dans le Journal système

			* La méthode laisse souvent un événement de type « Data Truncation » dans le journal
			* C'est un indice de modification de fichier, cohérent avec un remplacement
			* Procédure : → Voir Phase 11 – EventLogs (consulter le journal système)
			
			⚠️ Un Data Truncation NE prouve PAS à lui seul un remplacement → toujours croiser
			* Bonne pratique : comparer l'heure du Data Truncation avec l'heure de l'exécution suspecte.
			  Le remplacement étant immédiat, les deux événements doivent coïncider.
			
			(Réf. image : https://imgur.com/a/Qucx7Wi)

		────────────────────────────────────────────
		SYNTHÈSE DÉTECTION

			* Aucun indicateur ci-dessus n'est suffisant seul
			* La preuve = corrélation temporelle entre plusieurs indicateurs
			  (BAM + DPS + conhost + Data Truncation à la même heure)
			* Toujours corréler avec :
				- → Phase 5 (Prefetch)
				- → Phase 7 (BAM)
				- → Phase 10 (SRUM)
				- → Phase 12 (MFT)
				- → Phase 11 (EventLogs)

	Phase 22 – PE Injection (Portable Executable en mémoire) – DÉTECTION

		Objectif :
			* Détecter le chargement d'un PE (Portable Executable .exe ou .dll) en mémoire
			* Identifier un injecteur qui charge un cheat sans passer par le disque
			* Retrouver les signatures ou imports de l'injecteur en mémoire
			* Corréler l'injection avec les processus légitimes utilisant comme conteneur

		Rappel technique – Concept de PE Injection :
			* Au lieu d'exécuter un cheat via le chargeur standard de Windows (CreateProcess + LdrLoadDll),
			  un injecteur fonctionne ainsi :
				1. Crée ou ouvre un processus légitime
				2. Réserve de la mémoire dans ce processus (VirtualAllocEx)
				3. Écrit le PE en mémoire (WriteProcessMemory)
				4. Modifie le contexte d'un thread ou en crée un nouveau (SetThreadContext / CreateRemoteThread)
				5. Exécute le point d'entrée (Entry Point) du PE chargé manuellement
			
			⚠️ Il existe différentes méthodes de PE Injection ; elles ne fonctionnent pas toutes de cette manière.
			* Résultat du bypass : dans les traces système (BAM, Prefetch, etc.), 
			  ce n'est pas le cheat qui apparaît, mais le processus légitime utilisé comme conteneur.

		────────────────────────────────────────────
		Indicateur 1 | Retrouver l'injecteur via DIE (Detect It Easy)

			* Un injecteur classique (non obfusqué) peut contenir certains imports visibles
			* Outil requis : DIE (Detect It Easy) → Voir Tools Bible (ALL TOOLS SSA)
			
			Procédure :
				1. Ouvrir DIE
				2. Charger le fichier suspect (l'injecteur)
				3. Examiner la section « Imports »
				4. Chercher les imports typiques d'injection :
					- VirtualAllocEx
					- WriteProcessMemory
					- CreateRemoteThread
					- SetThreadContext
					- ResumeThread
			
			* Exemple de signatures d'injecteur : https://pastebin.com/MCKsUuKh
			* Tutoriel vidéo : https://www.youtube.com/watch?v=rY6-iBrct48 (Crédits : @shiya.py)

		────────────────────────────────────────────
		Indicateur 2 | Rechercher les signatures en mémoire RAM

			* Un dump RAM peut contenir des fragments du PE injecté
			* Procédure : → Voir Phase 2 – Mémoire & exécution ACTIVE (analyser avec FTK Imager et String64)
			
			* Mots-clés / signatures à rechercher :
				- Messages d'erreur du cheat
				- Chemins d'accès au cheat (%TEMP%, %APPDATA%, etc.)
				- Noms de fonctions importées
			
			* Exemple de signatures caractéristiques : https://pastebin.com/VBRyBxEE (Crédits : @shiya.py)

		────────────────────────────────────────────
		Indicateur 3 | Traces dans KernelLiveDump

			* Un dump Kernel live peut montrer l'injection en cours
			* Procédure : → Voir Phase 2 (analyser avec System Informer pour récupérer le dump)
			
			⚠️ Peut contenir des fragments de commandes ou d'imports non vus en mémoire utilisateur

		────────────────────────────────────────────
		Indicateur 4 | Corréler avec les processus conteneur

			* Identifier le processus légitime utilisé comme conteneur (javaw.exe, explorer.exe, svchost.exe, etc.)
			* Procédure : → Voir Phase 2 – Mémoire (lister les processus actifs avec leurs DLLs chargées)
			* Chercher des anomalies :
				- Processus légitime avec DLLs non standard chargées (AppData, Temp, Downloads)
				- Allocations mémoire inhabituellement grandes pour ce processus
				- Threads créés de manière suspecte (CreateRemoteThread laisse des traces)

		────────────────────────────────────────────
		Indicateur 5 | Retrouver l'injecteur via trace USN / Prefetch

			* L'injecteur lui-même doit être exécuté AVANT l'injection
			* Procédure :
				- → Voir Phase 4 – USN Journal (chercher la création du fichier injecteur)
				- → Voir Phase 5 – Prefetch (chercher l'exécution de l'injecteur)
			* Chercher un fichier récemment créé ou supprimé portant un nom banal ou aléatoire

		────────────────────────────────────────────
		Indicateur 6 | SRUM – corrélation temporelle avec le conteneur

			* Vérifier que l'injecteur et le conteneur ont été actifs au même moment
			* Procédure : → Voir Phase 10 – SRUM (analyser l'activité CPU / réseau)
			* Chercher :
				- injecteur + conteneur (javaw.exe) actifs simultanément
				- un pic anormal d'activité pendant que Minecraft était en cours

		────────────────────────────────────────────
		SYNTHÈSE DÉTECTION PE Injection

			* Aucune indicateur ci-dessus n'est concluant seul
			* Approche forensic :
				1. Chercher l'injecteur (DIE, USN, Prefetch)
				2. Vérifier le conteneur (processus avec DLLs suspectes)
				3. Corréler temporellement (SRUM, BAM, Prefetch)
				4. Analyser la mémoire pour les signatures (RAM dump, Kernel dump)
			
			* Une preuve solide = injecteur trouvé + conteneur identifié + signatures en mémoire + corrélation horaire
